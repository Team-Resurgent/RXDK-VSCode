Twenty-Five Tips for Improving Your Fill Rate

By Michael Austin and Mikey Wetzel, Xbox Advanced Technology Group

One of the challenges many games for the Microsoft® Xbox® video game system have is managing fill rate. Postprocessing effects, full screen antialiasing, and particle systems can bring the Xbox system to its knees. We've collected 25 tips for reducing fill rate that will help make your game look and perform its best.

0. Know the fundamentals

Two main factors affect fill rate: memory bandwidth and shader throughput. The total memory bandwidth for the Xbox is 6.4 gigabytes/second (GB/sec). The CPU is limited to around 1 GB/sec, leaving approximately 5.4 GB/sec for the GPU. The pixel shader operates on a 233-megahertz (MHz) clock. Each cycle, the pixel shader can do 4 pixels in parallel, giving a max throughput of around 933 megapixels/second (Mp/sec)—around 922 Mp/sec is the actual optimal value because of slight inefficiencies.

Both memory and shader throughput can constrain your fill rate; identifying which one is the constraint can help you target your optimizations. The ideal fill rate column in the Performance Investigator for Xbox (PIX) provides a good measure of how efficient your shader is. Many factors contribute to a lower actual fill rate, but if your shader is efficient and you're still experiencing a low fill rate, you are most likely memory bound.

1. Turn off alphablending when you don't need it

Alphablending can potentially cut the ideal fill rate in half on 32-bit render targets, due to the memory demands of its additional 4-byte color read per pixel. If a pixel shader always outputs opaque pixels, make sure to turn alpha blending off with this call:

SetRenderState( D3DRS_ALPHABLENDENABLE, FALSE );

2. Use alphatest for alphablended objects

Alphatesting can discard pixels with very small alpha values without doing a read or write to the render target. For particle systems and HUD elements, this can result in significant fill rate savings. The value 0x10 tends to work well as the ALPHAREF (the threshold alpha for pixels to be discarded by the alphatest), but different values can be tested for each effect.

After enabling alphatesting, it is important to verify that it is actually being used. Using PIX, look at the overdraw; if you see rectangular boxes around particles, alphatesting is not working properly.

A common reason for alphatesting to not work properly is that additive blending (also known as one:one blending) is being used with opaque textures, such as DXT1, where there is no alpha channel to test on. To fix this problem, simply copy the blue channel to the alpha channel in the pixel shader with an instruction such as:

mov r0.a, t0.b

To turn on alphatesting, use these render states:

SetRenderState( D3DRS_ALPHATESTENABLE, TRUE );
SetRenderState( D3DRS_ALPHAFUNC, D3DCMP_GREATER ); 
SetRenderState( D3DRS_ALPHAREF, 0x00000010 );

3. Use alphakill when appropriate

Alphakill is slightly different from alphatest in that alphakill discards pixels with very small alpha values on the basis of a texture lookup rather than as the result of the entire pixel shader calculation. If you have a complicated pixel shader, alphakill can save cycles by bailing when the texture fetch brings in an alpha value of exactly 0. For more information, see the Alphakill white paper.

Alphakill is set on an individual texture stage rather that the entire shader. To turn on alphakill for texture stage 0, use:

SetTextureStageState(0, D3DTSS_ALPHAKILL, TRUE );

4. Minimize z-buffer reads

Turn off z-buffer reads whenever possible. For 32-bit z-buffers, turning off reads saves 4 bytes per pixel of bandwidth with uncompressed data. Z-buffer reads are generally unnecessary when:

  • Drawing the skybox immediately after clearing the render target.
  • Drawing 2D HUD elements at the end of a scene.
  • Rendering to a projected shadow texture.

To disable z-buffer reads, use this call:

SetRenderState( D3DRS_ZENABLE, FALSE );

5. Minimize z-buffer writes

Similarly, turn off z-buffer writes whenever possible. For 32-bit z-buffers, this saves 4 bytes per pixel of bandwidth and preserves z-buffer compression. Z-buffer writes can be avoided in the same situations as listed for z-buffer reads. To disable z-buffer writes, use this call:

SetRenderState( D3DRS_ZWRITEENABLE, FALSE );

6. Maximize z-buffer compression

Depth buffer compression greatly reduces the memory bandwidth used by z-buffer reads and writes. Compression when the depth values are all the same, such as after a clear, is 8:1. Under normal circumstances, 4:1 is often possible. Since a 32-bit z-buffer can only be compressed if all of the stencil bits are zero, writing a zero into each of the eight stencil bits at the same time as you write the depth value can improve performance by 30 percent in some cases. The golden rules from the Depth-Buffer Compression and Performance Implications white paper are:

  • When you use the IDirect3DDevice8::Clear method on the depth buffer, clear the stencil buffer as well.
  • Always keep the stencil buffer cleared except when you are using it.
  • When you set the D3DRS_ZENABLE and D3DRS_ZWRITEENABLE render states to TRUE and aren't using a stencil, do one of the following: Set read-before-write by setting the D3DRS_ROPZCMPALWAYSREAD render state to TRUE, or always write a zero to the stencil buffer when you write to the depth buffer.

To clear the stencil buffer when you clear the z-buffer, use this call:

Clear( 0, NULL, D3DCLEAR_STENCIL | D3DCLEAR_ZBUFFER, 0x00, 1.0f, 0 );

To preserve a cleared stencil buffer when writing to the z-buffer, use this:

SetRenderState( D3DRS_ROPZCMPALWAYSREAD, TRUE );

7. Consider 16-bit buffers

Several shipping games have successfully used 16-bit buffers with great visual quality. Using 16-bit buffers cuts memory usage and bandwidth for reads and writes in half. This savings can be particularly useful for rendering environment maps or bloom effects where precision is not critical.

Note that the hardware performs best when the render target and the z-buffer are the same depth, so mix and match only after measuring real world performance using PIX.

8. Optimize pixel shaders

For fill rate limited calls (such as for the skybox or particle systems), make sure your pixel shaders are optimized. The Xbox hardware can do two combiners plus the final combiner and two texture fetches per clock cycle. Using a third combiner cuts your fill rate in half. Performing a third or dependent texture lookup also cuts your fill rate in half. There are many tricks to hand-optimizing pixel shaders that are outside the scope of this paper. If you want to learn more, the Xbox Advanced Technology Group (ATG) team would be happy to help; just send a PIX capture (which contains all the shaders and timing information) to your technical game manager or xboxds@xbox.com and request assistance.

For other useful tips, see the Pixel Shader Optimizing white paper.

9. Optimize interpolator use

The Xbox has a single hardware interpolator used for diffuse and specular interpolation in addition to the hardware texture interpolators. Using both diffuse and specular interpolation requires two cycles, cutting fill rate in half. However, if you only have one texture, you can often use the second free set of texture coordinates to do the specular interpolation and stay at maximum fill rate (though you do it through t1 not v1 in the pixel shader).

The white paper A Note on the Performance of Diffuse and Specular Interpolation provides details on how the interpolators work.

10. Reduce the number of passes

Whenever possible, try to collapse multiple rendering passes down to a single uber-pass. In most cases, the eight instructions and four texture fetches available in the pixel shader can do enough processing to combine two or more simpler rendering passes into a single pass. Combining all passes into one is especially important for ported games. Not only do you save CPU and GPU front-end (vertex processing) resources if you do one rendering pass, but fill rate is dramatically reduced. In one pass, 12 bytes of memory bandwidth are needed for every pixel (a 4-byte read and a 4-byte write to the depth buffer along with a 4-byte write to the back buffer). With two passes, 28 bytes of memory are needed for every pixel—12 bytes are needed for each pass, plus an additional 4-byte read for the alphablend.

One uber-pass can have a greater demand on the texture cache, as well as requiring more texture fetches and register combiners, but the benefits usually make it an obvious win.

11. Consider multisampling

Use 4× multisampling when the fidelity loss won't make a visual difference. This type of multisampling renders 16 pixels at a time instead of 4; each has an individual z value but replicated color.

Using 4× multisampling doesn't reduce memory bandwidth, but it cuts down on pixel shader usage. In the usual pipeline, the Xbox processes 4 pixels at a time. With 4× multisampling turned on, the Xbox still processes 4 samples but replicates the results to the surrounding 16 pixels (see the following diagram). The Xbox interpolates to positions A, B, C, and D and runs through a pixel shader pass, and then propagates the color information to the surrounding a, b, c, and d pixels.

c c d d
 C   D

Note that the z values and coverage information for these 16 pixels are not replicated but calculated individually.

Good uses of multisampling include:

  • Z-only rendering (such as to a shadow buffer).
  • Rendering solid colors.
  • Rendering low-fidelity images or images that lack high-frequency components such as bump-mapped water, ground, or sky.
  • Particle systems. Alphablending and overdraw tend to hide the loss of fidelity.

To turn on 4× multisampling, use this call:

SetRenderState( D3DRS_MULTISAMPLEMODE,  
  D3DMULTISAMPLEMODE_4X );

This type of multisampling does not result in a 4× reduction of fill. Actual results are difficult to measure but typically weigh in around a 20 percent reduction, which is still a very nice improvement. You can also use 2× multisampling for a smaller gain.

For more details, see the Antialiasing, Multisampling, and Supersampling white paper.

12. Experiment with the back-buffer scale

Back-buffer scaling can help improve fill rate by reducing the number of pixels drawn. This technique works by constraining the viewport to a smaller section of the back buffer, then scaling up the back buffer during the call to the IDirect3DDevice8::Swap method, where it's used as a texture and copied to the front buffer. Since full-screen antialiasing (FSAA) works the same way (using a larger buffer and scaling down), when using FSAA, back-buffer scaling is free—essentially allowing you to scale anywhere between 1× and true 4× FSAA.

Another variation on this trick is to render particles into a smaller off-screen buffer (such as 160×120 pixels), and then do an additive blend to the frame or back buffer. This technique dramatically reduces the fill requirements for particle systems, which are usually worst-case scenarios—alphablended and heavily overdrawn.

For back-buffer scaling (in this example to 50 percent), simply call:

SetBackBufferScale( 0.5f, 0.5f );

For more details, see the BackBufferScale graphics sample.

13. Use a smaller viewport

If you have a portal-based renderer, setting the viewport to the size of the portal can help save bandwidth and shader cycles. For example, if you are indoors, you can set the viewport to the visible door or window and clip out the pixels and triangles that aren't visible.

You can also shrink the size of the viewport you are rendering to if you know in advance that a large area of the screen will be overdrawn, for instance by an opaque HUD.

To change the viewport, use the following function call:

SetViewport( &NewViewport );

14. Draw a common-sense skybox

Unless your game is set in outer space, it's impossible to see below the horizon. Avoid drawing your skybox (and clouds, and so on) below the horizon to save fill rate.

15. Avoid redundant clears

All games start the scene by clearing the z-buffer, but only very rarely does a game need to clear the render target. One common excuse for clearing the render target is to reduce artifacts from cracks in the geometry, but it is better to fix the problem than to waste GPU resources every frame.

Check the number of Clear calls you see in a PIX capture. You should have one Clear call per render target; any others are probably redundant.

When you clear the z-buffer, don't forget to clear the stencil as well (see Tip 6, Maximize z-buffer compression).

For a 32-bit depth buffer, call Clear as follows:

Clear(0, NULL, D3DCLEAR_ZBUFFER | D3DCLEAR_STENCIL, 0x00, 1.0, 0);

16. Use back-face culling

Many games set the D3DRS_CULLMODE render state to CULL_NONE for many or all objects. Usually, the reason is to save polygons where the interior and exterior of an object are both potentially visible (trees with single polygons with both sides visible, for example). Sometimes, however, due to miscommunication between artists and programmers or a bug in the programming pipeline, art content is inconsistently set up and culling is left off to avoid "missing polygons."

Leaving culling off requires twice the fill rate on all polygons where it is done. It is better to fix the artwork to use proper back-face culling than to waste GPU resources. There are legitimate reasons for turning culling off (generally for situations that require two-sided lighting), but even in these cases it can be better to just duplicate the polygons to avoid mistakes.

You can identify missing back-face culling in PIX with the overdraw view. Mouse over a suspect part of the image, and check the overdraw count in the status bar. If your count goes up and down in increments of 2, then the polygons are drawn two-sided without back-face culling.

Set this once for your game, and don't look back:

SetRenderState( D3DRS_CULLMODE, CULL_CCW );

17. Render front to back

For opaque objects, rendering images front to back allows the z-test to discard pixels before running them through the pixel shader. This process saves clock cycles that can then go to shader instructions, texture fetches, and reads and writes to the back buffer for blending.

Rendering the skybox is an exception to this rule. If most of the skybox is visible, it's fastest to draw it first with z-buffer reads and writes turned off. If most of the skybox is not visible, it's fastest to render it last, using the z-test to reject most of the pixels.

18. Optimize triangle size with proper LODs

Use proper levels of detail (LODs) to achieve an ideal triangle-to-pixel relation. Triangles that are too small tend to be vertex bound, but with a lightweight vertex shader, they can hurt fill as well. The pixel hardware in the Xbox processes 4 pixels at a time in 2×2 quads, but small triangles have fewer than 4 pixels and can't fill the 2×2 quads. The maximum fill rate on the Xbox is 933 Mp/sec (4 pixels per clock cycle × 233 MHz clock rate). However, pixel sized triangles waste three of the pipelines, for an effective fill rate of only 233 Mp/sec. Roughly 25 pixels per triangle is a good target to aim for.

Use PIX to tell you if your LODs are proper (and therefore your triangle-to-pixel ratios). To do so, view the scene in wireframe mode. If any meshes look as if they were solid-rendered, you should adjust your LODs.

19. Use LODs on shaders, too

When thinking of LODs, don't stop at your geometry. For distant objects, rendering fidelity is less important. That eight-instruction, four-texture, bump-mapped object sitting off on the horizon can probably get by with a one-instruction, one-texture shader just fine. There's also a distance at which complex shadow algorithms can be reduced to a simple, static decal, and another distance at which shadows go away altogether.

The best engines try to determine what objects and characters in the scene deserve the most resources, using not just the distance but also other factors involving gameplay, situation, player psychology, and so on. When an object is less important, scale back the geometry and shader complexity.

20. Check overdraw

Check the overdraw view in PIX. Overdraw in opaque objects is wasted fill. This type of waste usually can't be entirely avoided, but problem areas should be apparent in PIX and should be cleared up. To improve the situation, change the order of your scene drawing to front to back.

Visibility testing can also help. In classic visibility testing, you render a bounding volume instead of a suspect object, and only render the full object if the visibility test for the bounding volume returns a nonzero pixel count.

21. Make genre-specific optimizations

Many games can benefit from genre-specific optimizations. For example, in racing games usually all four tires (and rims and brakes and so on) are drawn because they pass the camera cull test. Unless the car is above you in the air, however, you almost never see all four tires at once. The goal is to not draw what you know is not visible.

Some engines, such as portal engines, are already highly optimized for this goal, but many other engines can benefit from analyzing what is visible and what is not.

22. Write to tiled buffers

Tiling is a subtle but important consideration for fill rate performance. The Xbox rasterization hardware is optimized for writing pixels to tiled surfaces (that is, the memory is arranged specially in hardware to improve rendering performance). However, the texturing hardware doesn't deal well with tiled memory, so the question is when to use it.

In the back buffer and z-buffer, tiled memory is a must, and fortunately they are tiled upon creation. When you create your own off-screen render target, however, tiling is left up to you. By default, it is off.

To see whether you should turn tiling on for a render target, test. Measure your scene both ways using PIX. Generally speaking, the more a surface is rendered to, the more you should turn to tiling and improve rendering performance. For surfaces that are rendered once and referenced as textures often, leave tiling off.

To learn more about tiling, refer to the XDK documentation for the IDirect3DDevice8::SetTile method and the Tiling graphics sample in the XDK.

23. Optimize texture lookups

Which order and types of textures you use can make a big difference in performance. The FillRate tool, found in the ...\Tools\Xbox\Graphics\FillRate directory in your XDK setup, can help you choose the appropriate layout. Better yet, refer to the Texaddrops-alltests.txt file found in the same directory for every possible combination of texture addressing operations.

24. Optimize texture caching and use mipmapping

The texture cache in the Xbox is only 8 KB, which is enough space for storing one 32×64 texture at 32 bits per pixel. Larger textures and multiple textures competing for the texture cache space result in more cache misses, and thus more memory bus transactions to keep the cache filled.

Using smaller textures to reduce cache misses can reap huge rewards, which is the primary argument for using mipmapping; smaller levels make better use of the texture cache. Mipmapping increases the memory requirements of a texture by 40 percent, but the improved fill performance makes it worth it. Note, however, to not include mip levels that aren't needed. Character eyeballs do not need a 512×512 mip level, nor do 2D HUD elements need any mip levels at all.

In general, the Xbox memory controller reads from memory at 64 bytes per clock cycle. However, the implementation of the texture cache is such that only DXT compressed textures have a "cache fill rate" this high. All other textures are read into the cache at half-speed, or 32 bytes per clock cycle. Compressed textures are a win on many levels.

Palletized textures generally have the worst performance when filling the cache, as the Xbox has limited resources for decompressing palletized textures. This limited lookup rate can have an impact of many clock cycles. For more information, see the Xbox Palletized Texture Performance white paper.

Note that it is common for the occasional inappropriately sized texture to make it into a game. A good sanity check is often needed and can quickly be done by scanning the % texture bound column in PIX. If you find a suspect texture, ask yourself, "Is this a reasonable size? Can it be mipmapped? Is it compressed?"

Other tricks involve combining textures. One situation that comes up occasionally is the ability to put a scalar image (such as a specular map) in the alpha channel of a bump map, saving an entire texture fetch. You'll also get improved texture cache coherency and a faster shader to boot!

25. Check your numbers in PIX

PIX is a great tool for checking your fill rate; always check the ideal fill rate column and ask yourself whether you accept the result. Whenever the number is not roughly 933 Mp/sec, the reason why is somewhere in this white paper. Keep in mind that actual numbers do vary quite a bit due to memory bandwidth competition from the vertex processing and the CPU, and often a 466 Mp/sec or lower throughput is reasonable for more complex shaders.
