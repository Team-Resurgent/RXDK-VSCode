Xbox Watson Tool (XbWatson)

The XbWatson tool is actually a collection of tools that enable users to view debug output, display graphical asserts, and perform post-mortem analysis of a crashed Xbox application running on the Xbox console. The core feature of this tool is the ability for a running Xbox application to generate Dr. Watson–like dumps that can be used on an XDK development system to view a post-mortem stack trace of an application.

This tool will be primarily used by artists, content creators, and QA team members who are working with builds of a game but do not have access to or are not skilled with a debugger.

Debug Output Log

The debug output log is the primary window of the XbWatson tool. It displays any output generated from an Xbox title that gets sent across the debug channel. This includes: OutputDebugStrings, asserts, exception information, and crash logs.

The output log window is a simple Microsoft® Win32® window that displays text in an edit control. The user can scroll the contents of the window, select text, copy text to the Microsoft® Windows® clipboard, and save the output log as a file.

By default, XbWatson displays all debug output from both the running title and the Xbox kernel. To turn off the kernel debug output, from the Edit menu, unselect All Kernel Debug Output. Turning off kernel debugging output is a handy way to hide the noise generated by kernel debug messages so you can concentrate on your own title's output. Your title's debug output should be removed before submitting the title for certification, but kernel output normally does not impact certification.

Asserts

When an assert fires on an Xbox console (either from our kernel code or from the user's code by calling the CRT's assert function), it will appear on the Xbox development system in a consistent way with Win32 assert dialog boxes. The options presented after an assert are:

OptionDescription
Reboot Xbox Performs a quick reboot of the Xbox.
Break Causes the title to halt at an INT 3.
Continue Causes the application to continue running. If the assert is harmless, the program may continue without crashing.

Exceptions

Exceptions occur due to various problems in game code, including accessing illegal memory, breakpoints, and overflows. The options presented after an exception are:

OptionDescription
Reboot Xbox Performs a quick reboot of the Xbox.
Save Crash Dump Saves a file that developers can use to generate a stack trace. For more information on crash dumps, see Debugging Crash Dumps.
Continue Causes the application to continue running.

RIPs

A RIP is a fatal system software error that generally cannot be recovered from. The options presented after a RIP are:

OptionDescription
Reboot Xbox Performs a quick reboot of the Xbox.
Save Crash Dump Saves a file that developers can use to generate a stack trace.
Break Causes the title to halt at an INT 3.
Continue Causes the application to continue running.

Developer Experience

For a developer, the user experience for XbWatson is quite different. They will use the XbDumpLog command-line tool to generate a post-mortem stack trace. To use XbDumpLog, a developer will type the following at a command prompt:

xbdumplog -f [dumpfile] -p [path to .exe and .pdb file]

The output of this command is a decorated call stack with values for all of the function call's parameters; something such as:

USER32! 77e148fc()
WinMain(HINSTANCE__ * 0x00400000, HINSTANCE__ * 0x00000000, char * 0x0013237c, int 1) line 75 + 18 bytes
WinMainCRTStartup() line 198 + 54 bytes
KERNEL32! 77e992a6()

Usage Scenarios

  1. An artist for a sports title would like to alter some of the graphical media in his current build. He uses the file transfer tools to copy the media to his machine, alters it with his art tools, and then transfers it back to the Xbox. When he launches the game from XDK Launcher, the game hangs. He reboots his Xbox and runs XbWatson on his development system. When he launches the game again, he notices in the debug log window that the media file is corrupt. Shortly after, his game faults. He fixes the media file, re-transfers it to the Xbox, and launches the game without a hitch.
  2. A 3D artist on the sports title has made some modifications to one of the player meshes. He launches the game and it appears to hang. He fires up XbWatson and gets an assert that the number of vertices in the mesh is too high. He clicks Ignore and the game runs, but the frame rate is poor. He then returns to his modeling software and reduces the vertex and polygon count.
  3. A QA tester on the sports title always runs the latest "release" build of her game. When exceptions occur, they are often fatal. By running XbWatson on her Xbox development system, a debug dump is generated when crashes occur. She can then include this dump with her bug report. One of the game developers can subsequently use this dump to get a stack trace of the post-mortem application to help track down the problem.
