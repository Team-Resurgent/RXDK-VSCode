DmRegisterCommandProcessor

Registers a command handler.

DMHRAPI DmRegisterCommandProcessor(
  LPCSTR      szProcessor, 
  PDM_CMDPROC pfn
  );

Parameters

szProcessor
LPCSTR that represents the command processor.
pfn
Pointer to a command processor. Command processors must take the following form: 
   HRESULT __stdcall my_handler(LPCSTR szCommand, LPSTR szResponse, DWORD cchResponse, PDM_CMDCONT pdmcc);

Return Values

Returns XBDM_NOERR if successful, an error value otherwise.

Remarks

Whenever the command dispatcher receives a command in the form "handler!data data data", it will pass the entire command string (including the "handler!") to the command processor registered with a matching handler name. The command handler receives a buffer (szResponse) into which to place an optional response string. The value of cchResponse is the maximum size of the response. The return value of the command handler should be one of the XBDM_* error codes. Any other success codes will be translated into XBDM_NOERR and any other error codes will be translated into XBDM_UNDEFINED. If no response string is specified, the command dispatcher will construct a string based on the return code. If the return code is one of XBDM_MULTIRESPONSE, XBDM_BINRESPONSE, or XBDM_READYFORBIN, then the DM_CMDCONT structure is used (see below).

Generally, command handlers receive one string as input (szCommand, the complete command string) and are able to send one result code and response string (szResponse) as output. This is not sufficient for three types of command conversations: multiline textual response, receipt of binary data, or transmission of binary data. For each of these types of command conversations, a "command continuation" structure is used. Use of the various fields of the DM_CMDCONT structure vary slightly between conversation types; each type is outlined below, along with the uses of the DM_CMDCONT fields.

Note    There are many Xbox Title Libraries (XTL) functions that cannot be called safely from within a debug channel command handler. Any XTL function that is called that affects the value of GetLastError may cause the console to stop responding. Also, any return value from GetLastError may be incorrect. When developing a debug channel command handler, call as few XTL functions as possible.

Continued Transactions

There are three types of continued transactions supported. Each type requires use of a DM_CMDCONT structure, which is preallocated by the command dispatcher and which is passed into the command handler for initialization. Each type also requires that a specific result code be returned by the command handler to indicate that continued processing is necessary. The three conversation type return codes are XBDM_MULTIRESPONSE, indicating multiple lines of text response; XBDM_BINRESPONSE, indicating a binary response; and XBDM_READYFORBIN, indicating receipt of incoming binary data. Each of these three cases is described below.

XBDM_READYFORBIN

On exit from the command handler, the fields of the DM_CMDCONT should be set as follows:

  • HandlingFunction—the continuation function that is to be called when binary data is received.
  • DataSize—undefined.
  • Buffer—a buffer to hold incoming binary data. If this is NULL, the command dispatcher will provide a small (256-byte) buffer.
  • BufferSize—the size of the buffer indicated by Buffer.
  • CustomData—any value; this value will remain constant throughout the conversation unless changed by the continuation function.
  • BytesRemaining—the number of bytes of incoming data. This number may be zero.

On entry to the continuation function, the fields of the DM_CMDCONT will contain the following:

  • HandlingFunction—the continuation function that has been called.
  • DataSize—the number of bytes received. If this number is zero, then either the connection has been unexpectedly closed, or the initial value of BytesRemaining was zero, indicating that no data was expected. In either case that this number is zero, the continuation function should not expect to be called again.
  • Buffer—a pointer to the received data; unchanged unless it was previously NULL.
  • BufferSize—the size of the buffer, unchanged unless Buffer was previously NULL.
  • CustomData—unchanged.
  • BytesRemaining—the number of bytes of incoming data, including the bytes indicated by DataSize.

On exit from the continuation function, the fields of the DM_CMDCONT should be set as follows:

  • HandlingFunction—the continuation function that is to be called when binary data is received.
  • DataSize—undefined.
  • Buffer—a buffer to hold incoming binary data. If this is NULL, the command dispatcher will provide a small (256-byte) buffer.
  • BufferSize—the size of the buffer indicated by Buffer.
  • CustomData—as desired.
  • BytesRemaining—the number of remaining bytes of incoming data. If this number is zero, the command dispatcher will assume there is no more data to receive, and the continuation function will not be called again. If this number is zero, the return value of the function will be sent over the connection as the result of the transaction; if this number is nonzero, the return value of the function is ignored.

Note   If DataSize is zero on entry, BytesRemaining must be zero on exit to indicate that the binary transaction is complete. Failure to set BytesRemaining to zero in this case may result in loss of synchronization between the Xbox and the client.

Note   There is no provision in the protocol for early termination of the binary transfer due to errors, other than for loss of the connection. If an error is detected during receipt of the data, the handling function must continue to accept data. Failure to do so may result in loss of synchronization between the Xbox and the client.

XBDM_BINRESPONSE / XBDM_MULTIRESPONSE

Both binary response and multiline textual response are handled identically, except that at the end of multiline text data, the command dispatcher will send a dot on a line by itself to signify the end of data. When a binary response is sent, the command handler and the client must negotiate the size of the transfer.

On exit from the command handler, the fields of the DM_CMDCONT should be set as follows:

  • HandlingFunction—the continuation function that is to be called to prepare data for sending.
  • DataSize—undefined.
  • Buffer—optionally, a buffer to hold outgoing data. If this is NULL, the command dispatcher will provide a small (256-byte) buffer. No data will be sent prior to the first call to the continuation function, so no data should be entered into the buffer.
  • BufferSize—the size of the buffer indicated by Buffer.
  • CustomData—any value; this value will remain constant throughout the conversation unless changed by the continuation function.
  • BytesRemaining—any nonzero value.

On entry to the continuation function, the fields of the DM_CMDCONT will contain the following:

  • HandlingFunction—the continuation function which has been called.
  • DataSize—undefined.
  • Buffer—the buffer to hold outgoing data, unchanged unless it was previously NULL.
  • BufferSize—the size of the buffer, unchanged unless Buffer was previously NULL.
  • CustomData—unchanged.
  • BytesRemaining—unchanged, unless the connection was unexpectedly terminated. In that case, the continuation function will not be called again.

On exit from the continuation function, if there is data to be sent, the fields of the DM_CMDCONT should be set as follows:

  • HandlingFunction—the continuation function that is to be called to prepare data for sending.
  • DataSize—the number of bytes to be sent. If this is set to –1, then the data in Buffer will be sent as a NULL-terminated string, with a newline appended to the transmission.
  • Buffer—the data to be sent.
  • BufferSize—as desired.
  • CustomData—as desired.
  • BytesRemaining—any nonzero value.

If there is data to be sent, the function should return any success code. If the function returns XBDM_ENDOFLIST, the transfer will be assumed to be complete, and if the continuation was invoked with XBDM_MULTILINE, the command dispatcher will send a dot on a line by itself. If the function returns any error code other than XBDM_ENDOFLIST, the command dispatcher will forcibly terminate the connection without calling the continuation function again.
