Preparing to Send and Receive Messages

By default, messaging is in an "always on" state once the user logs on. At logon time, XOnlineLogon retrieves and caches a current snapshot of the user's message list, including the summary information. This list is managed internally by the messaging core and kept up-to-date locally behind the scenes as new message notifications arrive (when messages are added and deleted). Players can have a maximum of 125 messages. Each message summary is 64 bytes. Therefore, the messaging service allocates 8,000 bytes of memory for each player on the console connected to Xbox Live. It uses this memory to hold message summary information.

With the exception of retrieving details and attachments, and sending messages, all messaging API functions are synchronous. Work is done through the logon task handle on behalf of the messaging service in order to keep all message lists up-to-date automatically behind the scenes. This is done in an efficient and low-bandwidth manner for notifications and the downloading of message summary information. Titles may choose to disable the auto-retrieval of summary information if they feel that the extra bandwidth to upload summaries is too burdensome. To do so, call the XOnlineMessageSetSummaryRefresh function.

The messaging service uses notifications to alert the user to the presence of new messages. To receive these notifications, your title calls the XOnlineGetNotificationEx function. Notifications include the message IDs of the newest messages for a player. This enables the title developer to instantly grab the summary information from a message, in-game, if they wish to innovate in this area and process messages without requiring the user to pause the game. Embedded in each notification is a data member containing a set of flags. The flags indicate whether or not the title is required to display an in-game message notification icon. Titles will be required to consult this setting to determine whether to display a message notification icon in-game.
