Pixel Shader Optimizing

By Bruce Dawson, Xbox Advanced Technology Group

This paper demonstrates techniques for hand-optimizing of pixel shaders. Careful use of Xbox®-specific pixel shader instructions and the final combiner can frequently double the performance of pixel shaders, making this a worthwhile endeavor for expensive, fill-bound rendering operations.

Hand-optimizing of pixel shaders is important because the Xbox pixel shader assembler does no optimizations. It's all up to you.

This paper demonstrates the optimizing of a seven-stage (four-cycle) pixel shader into three or four stages (two cycles). The original shader is representative of shaders that we have seen in actual shipping games—similar optimization opportunities are not uncommon.

The purpose of the shader is not initially given in order to demonstrate that substantial pixel shader optimizing can be done, even if you don't know what the shader does!

Original Shader

xps.1.1              // use Xbox extensions

def c0, 1.0f, 0.0f, 0.0f, 0.0f
def c1, 0.0f, 1.0f, 0.0f, 0.0f

//--- Texture Stages ---
tex t0
tex t1
tex t2
tex t3

//--- Combiner Stage 0 ---
mul r0.rgb, t3.rgb, v1.a
+ mov r0.a, v1.b

//--- Combiner Stage 1 ---
mad r0, t2, r0.a, r0

//--- Combiner Stage 2 ---
dp3 r1, v1, c1

//--- Combiner Stage 3 ---
mad r0, t1, r1, r0

//--- Combiner Stage 4 ---
dp3 r1, v1, c0

//--- Combiner Stage 5 ---
mad r0, t0, r1, r0

//--- Combiner Stage 6 ---
mul_x2 r0, r0, v0

//--- Final Combiner ---
xfc zero, zero, r0, zero, zero, zero, r0.a

Turning the Shader into Pseudo-code

First, we go through the shader, one stage at a time, and rewrite it in a more readable form.

Cycle 1 Stage 0 r0.rgb = t3.rgb * v1.a
r0.a = v1.b
Stage 1 r0 = t2 * r0.a + r0
Cycle 2 Stage 2 r1 = v1 dot c1
Stage 3 r0 = t1 * r1 + r0
Cycle 3 Stage 4 r1 = v1 dot c0
Stage 5 r0 = t0 * r1 + r0
Cycle 4 Stage 6 r0 = 2 * r0 * v0
Stage 7 Unused
Final combiner result = r0

Turning the Shader into a Single Expression

Now we want to turn this into a single equation so that we can work on it more easily. In the equations below, the newly inserted expressions are underlined for emphasis. We start with the last equation:

r0 = 2 * v0 * r0

then insert the calculated value for r0 from stage 5.

r0 = 2 * v0 * (t0 * r1 + r0)

then insert the calculated value for r1 from stage 4.

r0 = 2 * v0 * (t0 * (v1 dot c0) + r0)

then insert the calculated value for r0 from stage 3.

r0 = 2 * v0 * (t0 * (v1 dot c0) + t1 * r1 + r0)

then insert the calculated value for r1 from stage 2.

r0 = 2 * v0 * (t0 * (v1 dot c0) + t1 * (v1 dot c1) + r0)

then insert the calculated value for r0 from stage 1.

r0 = 2 * v0 * (t0 * (v1 dot c0) + t1 * (v1 dot c1) + t2 * r0.a + r0)

Finally, insert the values for r0.rgb and r0.a from stage 0.

r0 = 2 * v0 * (t0 * (v1 dot c0) + t1 * (v1 dot c1) + t2 * v1.b + t3.rgb * v1.a)

That's not so bad.

It turns out that this shader implements a blend between four textures. The four channels of the specular color, v1, select between the textures, allowing an arbitrary mix between these textures. This shader can be used, for instance, to blend between four different terrain textures—perhaps gravel, road, grass, and dirt—to allow rendering a range of terrain types and transitions in a single draw call.

t2 and t3 are controlled by v1.b and v1.a, respectively. t0 and t1 are controlled by v1.r and v1.g, with dot products being used to isolate those channels—by setting c0 and c1 to pure red and pure green, respectively.

The result is then multiplied by the diffuse color and simultaneously multiplied by two.

Xbox Pixel Shader Hardware

Before we start optimizing, we need to understand the capabilities of the Xbox hardware. A combiner stage on the Xbox takes four inputs and can produce three outputs. The possible outputs and the instructions to invoke them are:

  • dot/dot/discard - xdd
  • dot/mult/discard - xdm
  • mult/mult/mux - xmmc
  • mult/mult/sum - xmma

These operations are what the Xbox GPU natively supports. Other instructions, such as mul, add, mov, and cnd are not supported directly; instead, they are implemented using these four instructions. So, for instance, this instruction:

mad r0, t0, t1, t2

is assembled to the same combiner stage definition as this:

xmma discard, discard, r0, t0, t1, 1 - zero, t2

Similarly, this trivial instruction:

mov r0, t0

is implemented using this instruction:

xmma r0, discard, discard, t0, 1 - zero, zero_sat, zero_sat

In other words, in order to move values from t0 to r0, the Xbox GPU multiplies t0 by one, then multiplies zero by zero, then adds the two products, and then finally writes the result to r0. This is a common technique used in the design of RISC floating-point instruction sets, so it's no surprise to see it showing up in the Xbox GPU.

While there's nothing wrong with using the mov instruction, it's clear that mov is not making full use of the hardware. If you can arrange your shader so that you're not always multiplying by one and zero, then you can get more work done in fewer stages, and maybe save a few clock cycles.

Optimizing for Xbox Combiner Stages

We can do the two dot products in one combiner stage using dot/dot/discard, storing the results in r0 and r1. We can then do a mult/mult/sum on the dot product results to blend in the textures, so we've calculated this:

r0 = t0 * (v1 dot c0) + t1 * (v1 dot c1)

in two combiner stages, like this:

xdd r0, r1, v1, c0, v1, c1
xmma discard, discard, r0, t0, r0, t1, r1

We can then calculate this:

r1 = t2 * v1.b + t3.rgb * v1.a

using another mult/mult/sum, like this:

xmma discard, discard, r1, t2, v1.b, t3, v1.a

Then we need to add the results in r0 and r1, multiply them by v0, and double the result. Another way of looking at this is we need to multiply both results by v0, add them, and double the results. So we need another mult/mult/sum, with an _x2 modifier:

xmma_x2 discard, discard, r0, v0, r0, v0, r1

The problem with this code is that it doesn't quite work. In the rgb pipe, the .b modifier has no effect. So in stage two, when we multiply by v1.b, we are actually multiplying by v1.rgb—which is not what we want. To avoid this, we have to use a previous instruction to copy v1.b to the alpha field of some temporary register. In this case, we can move v1.b to r1.a in stage 1, because r1 is temporarily unneeded. If we didn't have a spare register, we could use the alpha channel of a texture register or of v0 or v1.

The copy from v1.b to r1.a can be done in parallel with RGB operations and does not add any extra stages, because the Xbox pixel shaders have separate RGB and alpha pipelines. For any combiner stage, you can specify separate instructions for the RGB and alpha pipelines. You specify this by adding the .rgb modifier to the operands of one instruction, adding the .a modifier to the operands of the other, and putting a plus sign in front of the second instruction. The plus sign indicates to the assembler that you want the instructions paired; it's necessary because the pixel shader assembler will not automatically pair them.

The modified code is shown below:

Final Code: Four-stage Version

xps.1.1              // use Xbox extensions

def c0, 1.0f, 0.0f, 0.0f, 0.0f
def c1, 0.0f, 1.0f, 0.0f, 0.0f

//--- Texture Stages ---
tex t0
tex t1
tex t2
tex t3

; Calculate r0 = v1 dot c0, r1 = v1 dot c1
xdd r0, r1, v1, c0, v1, c1

; Calculate r0 = t0 * r0 + t1 * r1
; This gives us r0 = t0 * (v1 dot c0) + t1 * (v1 dot c1)
; It also copies v1.b to r1.a, for easier use in the
; next stage. Note that this means that r0.a will
; not receive any alpha from t1 or t0. The .rgb and .a
; instructions are paired, and execute simultaneously.
xmma discard.rgb, discard.rgb, r0.rgb, t0.rgb, r0.rgb, t1.rgb, r1.rgb
; Plus sign to indicate pairing.
+mov r1.a, v1.b

; Calculate r1 = t2 * v1.b + t3 * v1.a
; Note that v1.b is now stored in r1.a
xmma discard, discard, r1, t2, r1.a, t3, v1.a

; Now calculate r0 = 2 * (v0 * r0 + v0 * r1)
xmma_x2 discard, discard, r0, v0, r0, v0, r1

; This implements result = r0
xfc zero, zero, r0, zero, zero, zero, r0.a

Or, in more readable form:

Cycle 1 Stage 0 r0 = v1 dot c0
r1 = v1 dot c1
Stage 1 r0.rgb = t0.rgb.rgb * r0.rgb + t1.rgb * r1.rgb
r1.a = v1.b
Cycle 2 Stage 2 r1 = t2 * r1.a + t3 * v1.a
Stage 3 r0 = 2 * v0 * (r0 + r1)
Final combiner result = r0

We've shortened this shader from seven combiner stages (four clock cycles) to four combiner stages (two clock cycles). The only difference with the optimized version is that it ignores the alpha from t0 and t1. This shader was never used with textures containing alpha—the original shader ignored the alpha from t3—so this loss doesn't matter.

Using the Final Combiner

Our shader is now running twice as fast. However, it's worth noting that we're not using the final combiner for anything useful. The final combiner is overlapped with other operations, so it costs nothing to use it.

The final combiner does the following math:

rgbout = s0 * s1 + (1 - s0) * s2 + s3
alphaout = s6.b (or s6.a if the .a modifier is used)
prod = s4 * s5
sum = r0 + v1

You get to specify what s0 through s6 represent, using the xfc instruction. If we set s0 and s1 to v0 and r0, then we get our first multiplication. We have to set s2 to zero, since that part of the final combiner is no use to us. We can get our second multiplication using the special 'prod' register. 'prod' is the product of s4 and s5, and we can then specify it as s3. Thus, this final combiner:

xfc v0, r0, zero, prod, v0, r1, zero

almost perfectly takes the place of the final xmma. It isn't quite perfect because we're now missing the times 2. That can't be moved up into the other xmma instructions because this causes premature saturation; the intermediate results are 9-bit signed values with a range of only 1.0 (255 = 0x0FF) to –1.0 (–255 = 0x101).

This gives us a three-combiner-stage pixel shader. This three-stage shader won't run any faster than the four-stage version, but it leaves us with one combiner stage available for additional functionality. For instance, if we need the alpha from all four textures, then we could use this extra stage to copy v1.b to an alpha channel, instead of pairing this copy, to allow all calculations to operate on all four channels.

The optimization to three combiner stages is a tradeoff, since we can no longer multiply by two when multiplying by the diffuse color, so this optimization may not be the appropriate choice for this shader.

Another limitation of using the final combiner is that the inputs to the final combiner are always saturated. Any negative numbers will be clamped to zero.

Finally, D3D normally programs the final combiner to handle fog and specular. If you use the final combiner for other purposes, you will have to handle these effects on your own.

Final Code: Three-stage Version

xps.1.1              // use Xbox extensions

def c0, 1.0f, 0.0f, 0.0f, 0.0f
def c1, 0.0f, 1.0f, 0.0f, 0.0f

//--- Texture Stages ---
tex t0
tex t1
tex t2
tex t3

; Calculate r0 = v1 dot c0, r1 = v1 dot c1
xdd r0, r1, v1, c0, v1, c1

; Calculate r0 = t0 * r0 + t1 * r1
; This gives us r0 = t0 * (v1 dot c0) + t1 * (v1 dot c1)
; It also copies v1.b to r1.a, for easier use in the
; next stage. Note that this means that r0.a will
; not receive any alpha from t1 or t0.
xmma discard.rgb, discard.rgb, r0.rgb, t0.rgb, r0.rgb, t1.rgb, r1.rgb
+mov r1.a, v1.b

; Calculate r1 = t2 * v1.b + t3 * v1.a
; Note that v1.b is now stored in r1.a
xmma discard, discard, r1, t2, r1.a, t3, v1.a

; Use the final combiner to finish up.
; This implements result = v0 * r0 + prod, where prod = v0 * r1
; Thus, this implements result = v0 * r0 + v0 * r1,
; or result = v0 * (r0 + r1)
; Since we no longer have a times two, the diffuse colour should
; be brighter to compensate.
xfc v0, r0, zero, prod, v0, r1, zero

Or, in more readable form:

Cycle 1 Stage 0 r0 = v1 dot c0
r1 = v1 dot c1
Stage 1 r0.rgb = t0.rgb * r0.rgb + t1.rgb * r1.rgb
r1.a = v1.b
Cycle 2 Stage 2 r1 = t2 * r1.a + t3 * v1.a
Stage 3 Unused
Final combiner result = v0 * r0 + v0 * r1

Even if we did somehow optimize this down to two combiner stages, it would not improve performance. The four texture fetches take at least two cycles. Also, any shader that uses both diffuse and specular takes at least two cycles, because the Xbox GPU only has one general purpose interpolator. Because pixel shader performance is determined by the maximum of texture lookup cost, pixel shader cost, and interpolator cost, our maximum performance is two cycles.

Summary

The Xbox Development Kit, as of November 2003, contains the QuadLerp (four-way linear interpolation) sample, which can be used to compare the different versions of the pixel shader that are discussed in this paper.

When thinking about Xbox pixel shader optimizing, consider the following advice:

  • Don't optimize where it doesn't matter. Use PIX to identify draw calls that are expensive, fill bound, and not texture bound before hand-optimizing things. In most games, only a handful of pixel shaders actually need to be optimized.
  • The Xbox-specific instructions—xmma, xmmc, xdm, xdd, and xfc—open up much more pixel shader power, since they represent the full capabilities of the underlying hardware.
  • If you use v0 and v1 in the same shader, then your maximum performance is two cycles per pixel, because the Xbox GPU has only one general-purpose interpolator.
  • Don't forget about the PROD and SUM registers. They are crucial to optimal use of the final combiner.
  • If you are using fog, or otherwise need the final combiner, then some of the final combiner optimizations may not be available to you.
  • If you are using alpha in your textures then some of the parallel pipeline optimizations (separate rgb and alpha operations) may not be available to you.
  • Don't forget to look at the fillrate tool and TexAddrOps-AllTests.txt to find out what the maximum fill rate for your selection of texture ops is.
  • Roughly half of unoptimized shaders benefit just from getting rid of the one straggling odd instruction. Remember that pixel shaders take int((numStages + 1) / 2) cycles, so an odd number of stages is wasteful.
  • Break up geometry so that only relevant textures apply, instead of a works-in-all-cases shader. For instance, instead of using a four-texture shader to draw all of your terrain (blending between rock, grass, road, and dirt), draw those areas that are just one type of texture using a simpler shader. The savings in memory bandwidth and pixel shader instructions can be dramatic.
  • You can write to registers v0, v1, t3, t2, t1, and t0 and use them as temp registers. However, use this technique only when necessary because it can lead to unreadable code.
  • Beware of different results if the _x2 is moved to an earlier stage. Instead of "canceling out" the diffuse multiply, it will cause premature saturation.
  • When getting started writing pixel shaders, consider using the Xbox Preview Pipeline as an aid. You can view your models using the Preview Pipeline, apply the combination of effects that you want, and then use PIX to capture a frame and view the pixel shaders used.
  • The pixel debugger and pixel shader debugger in PIX are handy tools for debugging pixel shader problems.
  • You can make your code more readable by using #defines to give symbolic names to the constant registers.
  • Vertex shaders also have Xbox-specific instructions that can be used for maximum performance.
  • See the XDK Help for more information on pixel shaders.
