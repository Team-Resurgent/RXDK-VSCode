Registering for Arbitration

Before your title can register a session for arbitration, it must verify that the XNADDR structure for each Xbox console participating in the game have been shared with all of the other consoles. This is necessary to guarantee that the remaining players can continue to communicate in the event the match host should disconnect. Next, the match host must also call the XOnlineArbitrationCreateRoundID to get a round ID, also called a session ID, from the arbitration system.

If the game session is not a round in a competition, it passes the round ID to XOnlineArbitrationRegister. This registers the session. If, on the other hand, the game session is a round in a competition, the title calls XOnlineCompetitionSessionRegister rather than XOnlineArbitrationRegister.

Note    The triplet of the arbitration ID, the round ID, and the title ID (assigned before the game is released) uniquely identifies each arbitrated session on the system.

The match host next requests that each Xbox console register with the arbitration server immediately. All participating consoles must notify the match host when the registration is successfully completed. Each client registers for the session, providing the arbitration ID, the round ID, and the title ID. They must also specify the maximum session length and session flags indicating whether overtime is allowed, whether it's a team event, and whether it's a competition match. It if is a competition match, all registering consoles must specify the type (player- or publisher-managed; see Managing Competitions for details). Whether your title is registering the session with XOnlineCompetitionSessionRegister or XOnlineArbitrationRegister, the parameters (dwFlags and MaxSessionDuration) passed by all participants of the same session must be exactly the same.

In general, your game should use the dwMaxRoundSeconds parameter to set the maximum length of a round to about 5–10 minutes. Longer round lengths should not be used because they can cause problems for the arbitration service. The arbitration service does not check on the round until the maximum round length has expired. If, for instance, the round were to be somehow stopped 1 minute into a 30-minute round by a network outage, the arbitration service would not discover the problem for 29 minutes. Limiting round lengths to 5–10 minutes ensures that the service discovers such problems in a timely fashion. If you need longer rounds, you can get them by extending the round length as the round progresses. To do so, call XOnlineArbitrationExtendRound.

Note    Because of the time variations between Xbox consoles, your game should never set the maximum round length to a value of less than 3 minutes.

The flag XONLINE_ARBITRATION_REGISTERFLAG_TEAMS should be set when the game will update team stats or if the game is a team competition match.

The match host waits for everybody to notify it that registration is complete or waits a certain timeout period (5–10 seconds), whichever comes first.

The match host registers with the arbitration service and verifies the list of connected players with the list of registered players. It then disconnects from all the players who have failed to register and broadcasts the final registration list to all remaining players. This list of registrants should be the official list used during the game. Any Xbox that doesn't make it into that list must be disconnected from the game session.

If a client loses connectivity to the match host after it registers, but before game play starts, it should report to arbitration with an accurate "lost connectivity" report and no stats submission. Then it can proceed to find another game session.

Using the registration list as a reference, the clients can optionally perform a pairwise peer-to-peer QoS check. The goal of this optimization is to elect the client with best relative connectivity, QoS, and reliability values to be the game host (if the game is not pure peer-to-peer). The match host updates all clients with information about the new game host and cedes control of the game to the new game host. All clients then verify their connection to the new game host.
