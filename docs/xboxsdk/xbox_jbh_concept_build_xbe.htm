Building an XBE

This topic describes the process of compiling source code to an executable format that is compatible with the Xbox.

Creating the Project

The preferred way to create an Xbox executable (XBE) is to create a new Xbox Game project in the Microsoft® Visual C++® IDE and add your source files to that project. The Xbox Game project type contains all the switches and defines needed to build your XBE. The generated executable file will have "xbe" as its extension.

The Xbox SDK provides another useful project type—the Xbox Static Library project. The Xbox Static Library project type contains all the switches and defines needed to build a static library that can be linked with your executables. The generated file will have "lib" as its extension.

Note   With new releases of the Xbox SDK, project settings may change. To make sure that your projects are using the latest settings, create a new Xbox Game project and a new Xbox Static Library project with each new Xbox SDK release. You can then manually search for the changed flags and integrate them into your existing projects, or add the sources from those projects into the newly created projects.

Building the Project

Once you have properly configured the project and added your source files, you can build an Xbox title the same way you would build a Microsoft® Windows® application. See the Visual C++ documentation for more information about building applications from the Visual C++ IDE.

It is recommended that you link your projects with the debug Xbox Title Libraries (XTL). This is the default behavior of the Xbox Game project type.

Note   The IDE now automatically transfers generated XBEs to your Xbox console after the build is complete.

Consider these issues when building a new project:

  • If this is the first XBE to be built on your development system, you must specify the machine name of your Xbox console (see Using the Remote Debugger for instructions).

  • If this is the first time a particular project is being built, a dialog box will pop up asking for the remote path where you would like the generated XBE copied to. Enter the remote path and click OK. If you would like to change the remote path, you can edit the Remote executable path and filename field of the Project / Settings / Debug / Category: General dialog box. Remote paths will almost always start with "xe:\". Xbox samples are generally copied into their own subdirectory under "xe:\samples". But you are free to specify any location on the "xe:" drive.

    Notes   In debug mode, you are not prompted for the remote path. For a project named test, the IDE assumes a remote path of xe:\test\test.xbe. In general, "e:" refers to a local drive, and "xe:" refers to an Xbox drive. For remote paths specified in the IDE, "e:" and "xe:" are interchangeable.

  • The automatic copy feature can be disabled on a per-project basis, using the check box labeled Disable automatic copy of XBE file under Project / Settings / Xbox Image. However, there is a known issue that prevents this setting from being saved with the project file (.dsp). If you wish to disable the automatic copy feature, you will need to recheck this check box every time you open the project. This bug will be addressed in a future release of the Xbox SDK.

    Note   The automatic copy feature copies only the .xbe. If your .xbe relies on any media or data, include a post-build step similar to the post-build step in the samples—for example, "xbcp -r -y -d -t -f media\*.* $(RemoteDir)\media". See the File Transfer Tool (XbCp) for more information.
