Building Projects with Visual Studio .NET

The build commands in Visual Studio .NET have gone through significant changes from their equivalents in Visual C++ 6.0:

  • The biggest build change is that when you click Build on the Build menu, the IDE builds the entire Solution. This differs from Visual C++ 6.0, which would only build the Active project (displayed in bold) and all of its dependencies. The bold project is no longer considered "active" in Visual Studio .NET. While the bold project is still considered the startup project for the debugger, it no longer affects what is built.
  • If your solution consists of multiple projects, the project that is currently selected may be built using the menus. You can use the Build Project command to build the project and its dependent projects.
  • You cannot build a project without building its dependencies. This differs from the Visual C++ 6.0 Build (Selection only) commands.
  • There is no longer a node in the Solution Viewer showing that a project has a project dependency. Instead, there is a dialog box at Solution.Properties.CommonProperties.ProjectDependencies that provides this information.
  • Visual Studio .NET no longer has the concept of an Active Project for building. Many people liked having an active project that they could build, and then build all of its dependencies but nothing else. People liked this because some of the projects in their solution never needed to be built. Although there is no way to pin down one project as the active project, there is a way to prevent some of the projects in a solution from building. To prevent a project from ever being built, use the Configuration Manager dialog box, which you can get to by clicking Configuration Manager in the project settings dialog box. Deselect the build check box for each project that you don't want to participate in the build.
  • Visual C++ 6.0 developers often have difficulty adjusting to the selection behavior of the Solution Explorer. When you use the text editor to open a file in your project, the Solution Explorer makes your file visible by expanding the tree, selecting the file, and then scrolling it into view. If the file is not in one of your projects, the last project selected also has its project node selected. Many C++ developers are distracted by this, because the Solution Explorer "jumps" around the screen as the developer presses CTRL+TAB to switch through opened files. Unfortunately, there is nothing that can be done to change this behavior.
  • It is sometimes difficult to get the IDE to show you the full path to a file you have opened in the text editor. If the file is in your project, and you put the focus on the Solution Explorer, the property grid shows you the full path to the file. If the file is not in any of your projects, the IDE puts the file in the Miscellaneous Files project. By default, the Miscellaneous Files project is not shown in the IDE. If you enable showing the Miscellaneous Files project in the Solution Explorer, you will be able to see the properties of the files located in that project. To show the Miscellaneous Files project, navigate to Tools.Options.Environment.Documents, and in the Solution Explorer option, click Show Miscellaneous Files.
  • At the beginning of a build, the Output window is displayed and shows the progress of the build. At the end of the build, the Task list appears to show any errors or warnings that occurred in the build. The new Task list in Visual Studio .NET is not the best UI for C++ build output. To shut off the Task list, close the Task List window, navigate to Tools.Options.Environment.Projects and Solutions, and then deselect Task List if your build finishes with errors.
  • On the same Projects and Solutions page, there is an option to Save Changes on Build. There is a slight difference between Visual C++ 6.0 and Visual C++ .NET in that the project and solution are considered documents in Visual Studio .NET. This means that if you enable the Save Changes on Build option, when you press F5 the first time, the Save Changes on Build option forces you to save a copy of the solution file. In addition, enabling this option also saves changes to the project settings to disk.

Known Issues

This section lists known issues with using Visual Studio.NET to build Xbox projects.

  • If the XDK is installed on a system on which the Intel® pre-release compiler for Visual Studio .NET has been installed, Visual Studio .NET will crash whenever the user tries to open an Xbox project. In addition, the Intel pre-release compiler for Visual Studio .NET will not install on a system on which the XDK has been installed. This issue is fixed in version 7.0 of the Intel compiler. To obtain version 7.0 of the compiler, contact Intel Premier Support.

  • If you disable MAP file generation in Visual Studio, old MAP files are not deleted when the project is cleaned. As a result, you may receive the following compiler warning: 

    IMAGEBLD : warning IM1040: mismatched map file

    To resolve this compiler warning, do one of the following:

    • Enable MAP files: Project Properties->Linker->Debug->Generate Map File = "YES"
    • Add *.MAP to the extensions to delete on clean: Project Properties->General->Extensions to Delete on Clean
