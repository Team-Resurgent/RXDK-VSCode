Preparing for Xbox Submission

 

The last few miles of the march toward retail shelves are always busy and stressful. Sim-shipping with a brand new console just adds to the headaches. This paper pulls together a few issues that are easy to overlook or that have been poorly explained up to this point. It is focused mainly on the programming side of things and only briefly touches on the actual title submission process.  

Working with the System

Launch titles for the Xbox® video game system from Microsoft® will have had a relatively short period of time with final hardware compared with the length of development time of those titles. That time will be used mostly for ensuring the title actually runs on the hardware, tweaking to final performance, and other mission-critical steps. There are a couple of points worth noting.

Using the Startup Animation

Games will always be preceded by a startup animation, of which there are two variants. The first is a long version, played when the console is first turned on, that lasts six to eight seconds while the system is initialized, drives are spun up, and media is detected. The second is a shorter simple Xbox Logo screen. Both screens end with the same image: the Xbox slash logo and the Microsoft logo. When your title's main() is called, you can initialize Microsoft® Direct3D® and display your first animation, or you can take over the contents of the frame buffer to do more advanced transitions (wipes, fades, and so on). 

The last screen of the animation is always preserved by a call to PersistDisplay().  This means that this screen will continue to be displayed until the first time your application calls Present(). If you want to do a transition from that screen, you can retrieve the bits for that surface using the GetPersistedSurface() method. As the name implies, this method returns a pointer to the surface that holds the current video memory bits. If Present() has been called since PersistDisplay(), the pointer returned will be NULL. 

The surface that holds the persisted bits is not reference-counted. The interface should not be released after you are finished using it. The surface remains in memory only until the next Present() call. At that point, the pointer you hold to the surface becomes invalid. 

The only thing you need to do to this surface is copy the bits from the surface into your own buffer. The video signal is being scanned from this surface, so any attempt to lock this surface and modify the bits can result in tearing or other visual problems.

Note In the August 2001 Preview Release, a problem with the implementation of the persisted display memory causes that memory (approximately 1.25 MB) to be allocated in a random location in memory. Beginning with the August 2001 Final Release, this memory will always be allocated at the lowest virtual address.

Note New requirements regarding the specific handling of the logos on this screen will be included in an upcoming version of the Xbox Guide. 



The following sample code shows how to retrieve the bits from the persisted surface:

 

BOOL g_bDoTransition = TRUE;

IDirect3DTexture8  *m_pPT;

IDirect3DSurface8  *m_pPS;

IDirect3DSurface8  *m_pD3DPS;

 

// Create an empty surface to hold the persistent screen.

// Should be the same format as your application's back buffer.

m_pd3dDevice->CreateTexture(640, 480, 1, 0, D3DFMT_LIN_A8R8G8B8, 0, m_pPT);

m_pPT->GetSurfaceLevel(0, &m_pPS);

 

// Get the persisted bits, if possible.

GetPersistedSurface(&m_pD3DPS);

if(m_pD3DPS)

{

    // Got the bits, change resolution and format, if necessary.

    D3DXLoadSurfaceFromSurface(m_pPS, NULL, NULL,

                               m_pD3DPS, NULL, NULL,

                               D3DX_FILTER_TRIANGLE, 0);

}

else

{

    // Couldn't get the persisted bits, don't do transition.

    g_bDoTransition = FALSE;

}

 

// At this point, the persisted bits can be used to do a wipe or fade

// transition (or something more creative) to the game splash screen.

Launching Your Game 

Your game will be launched through one of the following mechanisms:

(1)   By the user turning on the console with the game disc already in the disc tray.

(2)   By the user inserting the game disc while in the Xbox Dashboard.

(3)   By the Xbox Dashboard returning control to the game after the game launched the Xbox Dashboard for a specific reason. 

The third mechanism is the only special case. The Xbox Dashboard will return the dwContext information originally passed to the Dashboard through the call to XLaunchNewImage. This context information can be used by the title to "remember" the reason and circumstance behind the original launch to the Xbox Dashboard. For instance, you could use the context to remember the last save game used by the user before launching the Xbox Dashboard to clean up space on a particular memory device. For more details on launching the Xbox Dashboard from your game, see the later section Launching the Xbox Dashboard from a Game. 

Testing For Final Hardware 

Several tools are available to help you develop and test your game against a platform that behaves as closely as possible to the behavior of the final retail console. 

The Xbox Game Disc Emulator 

Throughout most of the development cycle of games up to this point, there has been little choice but to use the Xbox console hard disk as the source for running your game. The XDK provided the XDK Launcher and the xE:\ drive semantics to make this as painless as possible. Although this method will continue to be supported, it has a number of significant drawbacks: 

•         It does not behave like the Xbox disc drive. Because it is just the hard disk, its seek, access, and streaming times are all much faster than will be achieved by your Xbox game disc in a retail console.

•         It interferes with your game's other uses of the hard disk. Because the development kit E:\ drive is physically part of the same device (and logical FATX partition) as your game's T:\, U:\, and Z:\ drives, it is difficult to optimize your use of those areas.

•         It is too small. Less space is available on the Xbox hard disk than on one Xbox game disc. Although a game build in itself may not take up that much comparative space, you may run into issues when you start using the hard disk for in-game caching and saves, with multiple builds on the machine, and, in some development houses, with multiple games on the same device.  

While you could use CD-RW or DVD-R media in your development kit to circumvent some of these issues, these media will not get you to final performance. Because game discs are multi-layered, neither format will take into account layer switches, and even the DVD-R media will not behave identically. Additionally, especially during heavy development, burning discs to test is not a particularly appealing option. 

With your Xbox Development Kit and host machine you have a better option, however. You can use the Xbox Game Disc Emulation software to mimic the behavior of the disc drive that will be available in the Xbox at launch. Although it requires adding additional hardware (shipped with your Xbox Development Kit) to your host machine, once set up, the emulator is simple to operate during development. In addition to alleviating the concerns mentioned above, the Game Disc Layout Tool can optimize the file layout of your game's build to take advantage of the nature of the Xbox disc drive. For example, you can push the most critical files to the outer (fastest) portions of the media, create duplicate copies of files to keep together files that are often read at the same point, and so on.  

For more details on using the Game Disc Emulation software, see the Xbox Game Disc Emulation System Setup and Upgrade section in the XDK Help.

Retail Mode 

Starting with the August release of the Xbox Development Kit (XDK), the recovery disk will have a new option that will allow you to "recover" your Xbox console into retail mode. The purpose of this mode is to allow you to test your game on a console that behaves exactly like a retail machine. When in retail mode, your XDK will: 

•         No longer have the E:\ drive or the XDK Launcher.

•         Use the Xbox Dashboard as the default title.

•         Fully enforce Xbox executable (XBE) certificate options (for additional details, see the later section Setting Executable Options).

•         Limit the amount of available memory to 64 MB.

•         No longer have debugger/developer support (including command-line tools, shader debuggers, and xbwatson).

Clearly, retail mode has significant limitations that will make it unsuitable at times during the development cycle. You can use the recovery disk to switch between development and retail modes as often as necessary. 

64-MB Mode 

Although not as close as retail mode above, the 64-MB mode, configurable on an XBE-by-XBE basis, is extremely useful in performance tuning your game for release. While the 128-MB footprint of the XDKs is useful for early development, it is good practice to start running your game with only 64 MB available as soon (and as often!) as possible. The 64-MB mode on Xbox is smart about allocations and actually allocates some memory above the 64-MB line to move debug and other non-retail code and data out of the way of your game. As a result, you game will run as close as possible to the actual memory configuration of the final retail box. 

For details on setting the 64-MB mode, see the XBE Image Builder Tool (ImageBld) topic in the Xbox Tools section of the XDK documentation.

Toggling the Default Title 

During development you will need to test the various scenarios that your game supports for launching to the Xbox Dashboard. By default, the Xbox development console always displays the XDK Launcher when no media are in the disc tray. The API to launch the Dashboard simply launches the default title of the console. On a retail Xbox, the default title is the Xbox Dashboard. However, on a development console, the default title is usually the XDK Launcher. As a result, you will need to change the consoles default title before doing scenario testing of these features. The xbsetcfg command-line tool allows you to switch between the XDK Launcher and the Xbox Dashboard for the default title. To set the Xbox Dashboard as the default title, use xbsetcfg -d. To set the XDK Launcher as the default title, use xbsetcfg -l. Note that while the development console is set to the Xbox Dashboard, you will be unable to use the XDK Launcher to run titles from the hard disk.

Setting Executable Options 

The Xbox development console is more lenient than the retail Xbox will be when determining whether or not to load a particular XBE. For the most part, this does not affect certification. If your game's ESRB rating is too high for the current console, the game is not supposed to run on its current media, or the game is meant to run only in a particular region, the retail Xbox will launch the Xbox Dashboard instead of your game and provide an appropriate error message to the end user. Your development console, on the other hand, will either pass you (because it defaults to signing your game for all regions, for instance) or just issue a debugger warning. In some cases, for instance, when testing locales or displaying sensitive game areas, it may make sense to directly manipulate and test these executable options. 

You can set specific ImageBld flags to manipulate these values from within your Microsoft® Visual C++® development environment. From within Microsoft® Visual Studio®, click on Projects in the menu bar, click Settings, click Xbox Image, and then add the following options in the Project Options dialog box:  

/TestId: <title id> The title identifier for this image. All executables with the same title identifier are considered part of the same game and are assigned the same storage areas when running on the Xbox. For instance, if you have a mainmenu.xbe and an engine.xbe, both will be able to write and read from the same data in T:\, U:\, or Z:\. A second game would not. Note that this option can also be set by clicking on the Title Info category from this page and setting it directly. You will be assigned an actual title ID for your retail game.

/TitleName: Name The name of the executable. The title name is used to create the friendly name of your game that is displayed to the user whenever they are looking at storage associated with your game. This switch allows only one entry. If you want text that is localized into different languages based on the current global console language setting, use the /TitleInfo switch instead. You can set this name from the Title Info category on this page directly. 

/TitleInfo: <Path> Specifies the path to the file containing the title information localized into as many of the six supported languages as the title supports. Note that you must have a [default] section in this file to pass title certification. You can set this path name from the Title Info category on this page directly.  

/TitleImage: <Path to XPR File> The path to your title representative image, displayed in the large orb from within the Xbox Dashboard. You can also set this from within the Title Info category on this page. For details on creating this image, see "Dashboard Metadata" under "Working with Xbox Executable Models" in the "Developing for Xbox" section of the XDK Help.

/DefaultSaveGame: <Path to XPR File> As above, this is the path to your titles default save game image. This image is displayed whenever you create a save game (by using the XCreateSaveGame function) and do not copy a specific save game image to the resulting save game directory. In general, your title will look better from within the Xbox Dashboard if each save game has a unique image (such as a screen shot). This image can be set from the Title Info category on this page. For details on creating this image, see "Dashboard Metadata" under "Working with Xbox Executable Models" in the "Developing for Xbox" section of the XDK Help.

Approved Versus Unapproved Libraries 

At certification, your game must link only with approved libraries. Approved libraries are defined on a month-by-month basis and will be documented in each month's XDK. In general, release versions of the Xbox Title Libraries are approved. Debug versions, special debug libraries, and beta or preview libraries of new functionality are unapproved. 

Starting with the August 2001 release of the Xbox Development Kit, ImageBld will warn you if you attempt to link with unapproved libraries. In the build window of Visual C++, if you link with the debug keyboard library, you will see the following output: 

IMAGEBLD : warning IM1029: library xkbdd.lib is unapproved. 

Additionally, if you use certain libraries that require additional approval (for instance, the networking support), you will receive this message: 

IMAGEBLD : warning IM1028: library xnets.lib may not be approved.  

Testing for Certification

Several tools are available to help you test your game against many of the Technical Certification Requirements (TCRs). As time and feedback permits, we intend to make more such tools available. When preparing your test plan, use the Xbox Guide as your main resource. The Guide is the overriding source for Technical Certification Requirements. In addition to the actual requirement text, many entries also provide comments and explanations to help clarify the objective. To stay on top of any changes in requirements, refresh your copy of the Xbox Guide every month.

Additionally, the Technical Certification Game is a (not very fun) game designed to provide a code sample of many of the certification requirements. The sample is explained in detail in the XDK Help topic "Xbox Game Certification" under the "Developing for Xbox" section. Of particular interest from a development point of view are the source code comments describing what certification requirement is being solved by a particular snippet of code. For example, the following code snippet (from loadsave.cpp) is designed to meet three different certification requirements (1-10, 3-11, and 3-12).

    // TCR 1-10 Memory Unit Maximum Capacity   

    // Get the total and used bytes.

    ULONGLONG qwTotalBytes;

    ULONGLONG qwUsedBytes;

    ULONGLONG qwFreeBytes;

    if( GetSize( qwTotalBytes, qwUsedBytes, qwFreeBytes ) )

    {

        // Convert to blocks.

        // TCR 3-11 Space Display

        DWORD dwBlockSize   = CXBStorageDevice::GetBlockSize();

        DWORD dwTotalBlocks = DWORD( qwTotalBytes / dwBlockSize );

        DWORD dwUsedBlocks  = DWORD( qwUsedBytes / dwBlockSize );

        DWORD dwFreeBlocks  = dwTotalBlocks - dwUsedBlocks;

 

        // Format with thousands separators.

 

        // TCR 3-12 Maximum Number of Blocks

        // Any amount over 50,000 blocks is formatted "50,000+".

        if( dwTotalBlocks > MAX_BLOCKS )

            lstrcpyA( strTotalBlocks, strMAX_BLOCKS[ CXBConfig::GetLanguage() ] );

        else

            CXBConfig::FormatInt( dwTotalBlocks, strTotalBlocks );

 

        if( dwFreeBlocks > MAX_BLOCKS )           

            lstrcpyA( strFreeBlocks, strMAX_BLOCKS[ CXBConfig::GetLanguage() ] );

        else

            CXBConfig::FormatInt( dwFreeBlocks,  strFreeBlocks );

 

        // TCR 3-11 Space Display

        // Graphical representation must only account for the high 50,000

        // block range. If the device has more than 50,000 blocks, adjust

        // total to 50,000, and scale "used" into the 50,000 section.

        if( dwTotalBlocks > MAX_BLOCKS )

        {

            dwTotalBlocks = MAX_BLOCKS;

            dwUsedBlocks = ( dwFreeBlocks > MAX_BLOCKS ) ? 0 :

                                                    MAX_BLOCKS - dwFreeBlocks;

        }

 

        fPercentUsed = FLOAT( dwUsedBlocks ) / FLOAT( dwTotalBlocks );

    } 

XDK Launcher Tools 

Starting with the July 2001 release, the XDK Launcher contains a set of tools designed to interact with the Xbox console to help you test your title in various situations. These tools can be accessed by pressing the X button while in the main menu.  

Launching the Xbox Dashboard from a Game 

When a game runs, it has complete control of the Xbox console. However, a game still needs to provide the functionality to use the Xbox Dashboard for certain tasks (for instance, to free other games' data from the hard disk).

To launch the Dashboard from a game, simply create an LD_LAUNCH_DASHBOARD structure (defined in xbox.h) and fill out the parameters as described below. At that point, simply call XLaunchNewImage (NULL).

For instance, the technical certification sample game uses the following code snippet to boot to the Xbox Dashboard to clean up space on a device:

// TCR 3-4 Cleanup Support

// Launch to the Dashboard Memory Area to clean up.

if( pGamepad->bPressedAnalogButtons[ XINPUT_GAMEPAD_B ] )

{

DWORD dwBlocksNeeded = dwBytesRequired /

CXBStorageDevice::GetBlockSize();

 

// Reboot to Dashboard.

LD_LAUNCH_DASHBOARD LaunchDash;

LaunchDash.dwReason = XLD_LAUNCH_DASHBOARD_MEMORY;

LaunchDash.dwContext = 0;

LaunchDash.dwParameter1 = DWORD( HardDisk.GetDrive() );

LaunchDash.dwParameter2 = dwBlocksNeeded;

 

XLaunchNewImage( NULL, (PLAUNCH_DATA)(&LaunchDash) );

 

// We never get here.

}

 

You can boot to the Xbox Dashboard using the following parameters:

Memory: User is launched to the memory details screen for the device specified.

•         dwReason = XLD_LAUNCH_DASHBOARD_MEMORY

•         dwParameter1 = Drive letter to clean space on. (Either T or U for the hard disk or any mounted memory unit drive letter.)

•         dwParameter2 = Number of blocks to free on the device.

Settings: User is launched to the Settings Area.

•         dwReason = XLD_LAUNCH_DASHBOARD_SETTINGS

•         dwParameter1 = Specific setting screen to launch.

o        XLD_SETTINGS_CLOCK 0x01

o        XLD_SETTINGS_TIMEZONE 0x02

o        XLD_SETTINGS_LANGUAGE 0x04

o        XLD_SETTINGS_VIDEO 0x08

o        XLD_SETTINGS_AUDIO 0x10

Music: User is launched to the Music Area.

•         dwReason = XLD_LAUNCH_DASHBOARD_MUSIC

Main Menu: User is launched to the initial Xbox Dashboard screen.

•         dwReason = XLD_LAUNCH_DASHBOARD_MAIN_MENU

When the user is finished with the task in the Xbox Dashboard and is returned to the game, you can retrieve the dwContext field stored in the LD_LAUNCH_DASHBOARD structure by calling XGetLaunchInfo. When returning from the Dashboard, the dwLaunchDataType will be set to LDT_FROM_DASHBOARD, and the context can be retrieved from the LD_FROM_DASHBOARD structure.

Note on Testing Launching to the Dashboard

Due to the nature of the launch process, the Xbox Dashboard will attempt to launch your title from the disc drive. During development, if you launch your title from the XDK hard disk, the Dashboard will not find your title when rebooting. To circumvent this limitation, always use the game disc emulator or hard media when testing. Additionally, before testing this feature, see the earlier section Toggling the Default Title on setting the Dashboard to be the default title.

Global Xbox Settings

The Xbox console has a variety of global settings usable by your game to determine the connected state of the console (which AV packs, for instance), the environment of the user (Dolby® Surround sound, HDTV, and so on) and any user preferences. Your game should use these settings to make intelligent decisions when determining how to play on a particular system. See the settings certification requirements in the Xbox Guide for specific details on how these settings and your game will be tested in certification.

How Do the Dashboard Video Settings Affect My Game? 

The user has the option to set a number of video settings in the Xbox Dashboard. These video settings influence the set of video modes that your game can select. The user can set the following settings: 

For all AV packs:

Normal/Letterbox/Widescreen (mutually exclusive) 

If the Xbox HDTV AV Pack is installed:

Enable 480p

Enable 720p

Enable 1080i 

If the console AV region is XC_GAME_REGION_RESTOFWORLD:

Enable PAL-60 

The Normal/Letterbox/Widescreen setting is used to specify whether the user's TV is a widescreen display. If either Normal or Letterbox is selected, a game should not attempt to select a widescreen display mode. If Widescreen is selected, a game may choose to select a widescreen video mode, but it is not required to do this. A game selects a widescreen video mode by setting the D3DPRESENTFLAG_WIDESCREEN flag in the flags field of the D3DPRESENT_PARAMETERS structure that is specified when the Direct3D device is created. If widescreen is not enabled by the user, attempting to create the device with this display mode will fail. 

A widescreen video mode activates a special signal to tell the TV to switch into widescreen mode. Some TVs do not recognize this signal, and the user will be required to manually switch the TV. If your game selects a widescreen mode, it will also have to adjust its transform matrix to provide a 16:9 display aspect ratio rather than the more common 4:3 aspect ratio.

The HDTV settings are enables and they are not mutually exclusive. A user enables the 480p setting to indicate that his TV supports 480p (progressive scan). This does not require a game to support 480p. The same holds true for 720p and 1080i. If the user has enabled 480p and the game is not doing field-rendering, the game should select 480p rather than 480i (interlaced) because the picture quality is better. A game selects a the 480p rendering mode by specifying the D3DPRESENTFLAG_PROGRESSIVE flag in the flags field of the D3DPRESENT_PARAMETERS structure that is specified when the device is created. The back buffer height must be 480, and the back buffer width must be either 640, 704, or 720. If the back buffer height is 720, the 720p mode is selected (if enabled).

If the console supports PAL (AV region is XC_GAME_REGION_RESTOFWORLD), the user has the option of enabling PAL-60, indicating that his TV supports this display mode. If this option has been set, the game can select a video mode with a refresh rate of 60 Hz. This is done by specifying the FullScreen_RefreshRateInHz field to be 60 in the D3DPRESENT_PARAMETERS structure that is specified when the device is created. Normally this field is 50 Hz for PAL modes. The Xbox video hardware will set up the proper output signal for PAL-60. The only difference that the game will notice is that the vertical blank will occur 60 times per second rather than 50. A game that runs well at 60 Hz will not likely have performance problems running at 50 Hz. However, some content is authored with a specific playback frame rate in mind. For example, video for cut scenes may be authored to be decompressed and displayed at 60 Hz. Or motion capture sequences may be authored for 60 samples per second.

There are several approaches for dealing with this type of content. The best solution is to author two versions of this content. One should be ideal for playback at 50 Hz, and the other should be ideal for playback at 60 Hz. The game can choose the appropriate version at startup time when it determines the type of display. There will likely be ample room on the game disc to hold both versions of the content. This solution offers the best performance and quality. Another solution is to interpolate the content at run time based on the time between frames. The system time is read at the start of each frame, and the animation or video is interpolated based on that time. This solution may be more difficult with some types of content than with others, and it does cost some run-time performance. It does have the added advantage that it properly handles cases in which the frame rate is changing due to system overload. The third solution is to drop frames of video or motion capture samples. If one frame in every six is dropped on content authored for 60 Hz, it will play back at the proper rate for 50 Hz. This solution doesn't cost much performance but can result in noticeable quality degradation. 

It is easy to retrieve the settings described above using the XGetVideoFlags() function, which returns a DWORD that contains the flags listed below. If the bit is set, that flag is enabled. The flags are as follows:

#define XC_VIDEO_FLAGS_WIDESCREEN 0x00000001

#define XC_VIDEO_FLAGS_HDTV_720p 0x00000002

#define XC_VIDEO_FLAGS_HDTV_1080i 0x00000004

#define XC_VIDEO_FLAGS_HDTV_480p 0x00000008

#define XC_VIDEO_FLAGS_LETTERBOX 0x00000010

#define XC_VIDEO_FLAGS_PAL_60Hz 0x00000040 

Because the valid video modes are also influenced by these flags, a game can simply ask for all of the valid video modes using the EnumAdapterModes() method. The number of valid modes can be retrieved with the GetAdapterModeCount() method. As long as the game selects one of the video modes enumerated in this manner, the device creation will not fail due to an invalid video mode. 

Back Buffer Resolution and Video Modes

Your game can always select a back buffer resolution of 640 × 480, which will work with NTSC, PAL, and HDTV display modes. The native resolution for NTSC is either 640 × 480 or 720 × 480. The native resolution for PAL is either 640 × 576 or 720 × 576. The native resolution for 480p is 720 480p. The native resolution for 720p is 1280 × 720. Finally, the native resolution for 1080i is 1920 × 1080i.

If your back buffer size is specified to be 640 × 480, the front buffer will be scaled to match the native resolution for the selected display mode listed above. To prevent this scaling, simply allocate a back buffer that is the same size as the native display mode. For example, for PAL output, allocate a frame buffer that is 640 × 576 pixels, and no scaling will occur. 

Special Video Modes

Two special video mode attributes may be applied to the modes described above. If the D3DPRESENTFLAG_FIELD flag is specified, the back buffer size is assumed to be half the vertical height of the output display. The output format is also defined to be interlaced. In this mode, the even lines of the display are rendered during one field and odd lines are rendered during the next field. The flicker filter is disabled.  Field rendering can reduce the fill-rate and bandwidth requirements of a game substantially, but because there is no flicker filter, display quality can also be significantly degraded. Note that field rendering is supported when the Xbox HDTV AV Pack is installed because 480i is allowed. 

The second video mode attribute that may be specified is D3DPRESENTFLAG_10X11PIXELASPECTRATIO. This mode increases the black space on the right and left sides of the screen so that more of the rendered pixels are squeezed into the active portion of the screen. If you choose to use this mode, you will need to adjust the transform matrix for your game to accommodate the non-square pixels in the display (slightly taller and thinner 1:1.1).

How Do the Dashboard Audio Settings Affect My Game? 

The user has the option to set a number of audio settings in the Xbox Dashboard. These audio settings determine both how audio data for gameplay and DVD movie playback is output from the Xbox. 

For all AV packs:

Mono/Stereo/Dolby Surround.  (Mutually exclusive settings) 

If HDTV AV Pack, Advanced AV Pack, or Advanced SCART Connector is installed:

Enable Dolby Digital (y/n)

Enable DTS (y/n) 

The Mono/Stereo/Dolby Surround setting is used to configure how two-channel data is output from the box through the analog left and right audio outputs and the digital audio output if Dolby Digital is not enabled. The three options are mutually exclusive.

Mono

Mono should be set if only one of the two output channels is physically connected to an audio source. The most common scenario for this would be a television that has only a single analog audio input. In Mono mode, Microsoft® DirectSound® mixes all the speaker sources to a single mono stream and outputs over both analog and digital outputs.

Stereo (Default)

Stereo should be set if the Xbox console is connected to a system with only two speakers (left and right) whether digital or analog. In Stereo mode, data sent to the left rear mixbin is mixed into the left front channel, data from the right rear mixbin is mixed into the right front speaker, and data from the center channel mixbin is mixed into both left and right speakers, as is the LFE mixbin.

Dolby Surround

Dolby Surround should be set if the users Xbox console is connected to a Dolby Pro Logic sound system with surround speakers. In Dolby Surround mode, data from the six speaker mixbins are sent through the Dolby Surround encoder.

The Difference Between "Stereo" and "Dolby Surround"

Because Dolby Surround is in fact a stereo format, the question might come up, "How is this different from stereo?" There are several reasons for this distinction. 

•         The Dolby Surround encoder takes the inputs from left, right, center, left surround and right surround to make a stereo signal. To make this five-channel input (the Dolby Surround encoder drops the LFE channel) into a two-channel signal, it takes the left and right surround channels, mixes them together, low-pass filters the result, and encodes the signal to come out of the surround channel on a Dolby Pro Logic system. It then mixes this signal into the left and right outputs. If the user has a Pro Logic Surround system with surround speakers, this is ok, because it causes the sounds routed to the rears to actually emanate from the surround speakers. However, when listening over only two speakers, the sounds placed in surround mixbins or
3D sounds behind the listener are unnaturally low pass filtered. So having a separate "stereo" setting allows these rear-channel sounds to be played at full bandwidth and fidelity. 

•         A second reason has to do with the 3D Head Relative Transfer Function (HRTF) processing. When in Surround mode, the 3D processor uses a separate panning algorithm, placing some sounds in the surround channels (so people with surround systems hear 3D sounds from the surrounds when the source is behind the listener). If there are only two speakers, the HRTF processor only places data in the front speakers. This provides a much better 3D effect when listening over just two speakers. 

•         A third reason is that, as noted, the Surround processor combines the left and right rear sounds together. So, if there were a stereo ambience placed in the rears, it would come out of the rears (as a mono signal) when in Dolby Surround mode. But in "stereo" mode, the sounds will maintain their stereo separation and merely be mapped to the left and right fronts. So for those reasons (filtering of rear speaker data when going through surround encoder,  optimized 3D HRTF processing, and the ability to keep stereo separation) there is a distinction made between "Stereo" and "Surround."

Dolby Digital

The Dolby Digital Enable (y/n) setting is used for two things: First, it sets the format of the digital output during gameplay to Dolby Digital; that is, the real-time Dolby Digital encoder is used to process all the sounds, and the output of the encoder is played out through the Digital output. 

Note If Dolby Digital is set to Enable, the Mono/Stereo/Dolby Surround setting is automatically set to Dolby Surround.

Second, the setting sets the DVD movie player to output raw Dolby Digital data via the digital out. 

DTS

The DTS Enable (y/n) setting is used only for DVD movie playback. It has no effect on game audio. 

Headphones

There is no Xbox Dashboard setting for headphones. Because headphones is not considered a permanent configuration, it was not included as part of the Xbox Dashboard. There is a DirectSound interface, IDirectSound::EnableHeadphones(bool), that can allow a game to put Xbox into headphone mode. EnableHeadphones() was added to DirectSound because the
3D HRTF processes perform slightly different work depending on whether the user is listening over headphones or speakers. Note that if you are not using the 3D libraries (DirectSound3D parameters or Microsoft DirectMusic 3D audiopaths), there is no point in using this call.

Headphones mode does the following:

         Forces output to be stereo and disables crosstalk cancellation for optimal headphone listening. This does disable multichannel output; all sounds will come through only the front left and right speakers, even with Dolby Digital enabled.

         Turns off "crosstalk cancellation" from front speakers. This optimizes HRTF for headphone listening.

Ideally, the ability to select headphone or speaker listening (EnableHeadphones(TRUE) or (FALSE)) would be set via an in-game audio options screen, though this is not a requirement. The DirectSound sample, Play3DSound, shows how to use this function.

Other Particulars

When an RF adapter (which can only output monaural audio) is connected to the Xbox, the control panel setting is overridden and forced into mono mode.

Tailoring Audio

By default, DirectSound (and DirectMusic) will try to do the "right thing" when placing 3D sounds and sounds placed in specific mixbins according to the Xbox Dashboard setting. However, a game can query the Dashboard audio settings directly by using XAudioGetSpeakerConfig(). The game can then change how audio data is routed to individual mixbins, if desired. 

User Education

It is a good idea to place some basic information about audio system configuration in your game manual to remind the user of the importance of setting their audio settings. Example text is included below:

The Xbox provides real-time interactive multi-channel game audio supporting Dolby Digital, Dolby Surround and 3D HRTF processing. For the best audio experience, it is important that your Xbox be configured to match your particular sound system's capabilities. To do this, in the Settings Area, select AUDIO. If your Xbox is connected to two speakers, select Stereo. If you have a Dolby Pro Logic receiver with surround speakers, select Dolby Surround. If you have an Xbox AV connector that provides digital audio out and you have a Dolby Digital receiver with 5.1 speakers, enable Dolby Digital. Following these simple steps will help you get the best audio experience possible for Xbox gameplay. Note that an AV pack that supports digital audio out, such as the HDTV Pack, Advanced AV Pack, or Advanced SCART Connector, is required for Dolby Digital.

Differences Between Retail and Pre-Certification Games

In the process of certifying and preparing your game for duplication, some minor changes will have been made.

All certification fields (rating, region, and so on) will have been stamped with their final values.The key used to do save game verification will be final. Save games created by the pre-certification version will not pass the XCalculateSignature functions.

Your game will now run on a retail console and will not run on a development console.

If your game supports system link play, it will no longer communicate with the pre-certification version.
