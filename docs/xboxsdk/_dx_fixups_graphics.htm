Fix-ups

Because a push-buffer resource is a direct recording of the Xbox GPU instruction stream, there is no Direct3D object semantic data recorded along with the push-buffer. There are no Direct3D objects recorded in the push-buffer, only physical addresses. Consequently, when a push-buffer is loaded from disk and registered, all resources that were originally referenced by the push-buffer must be explicitly fixed-up to account for potentially new physical addresses of the resources involved.

There are two types of fix-ups that have to be applied to push-buffers: one-time fix-ups and per-call fix-ups.

One-time fix-ups update push-buffer references to resources, such as textures, which are unchanged between calls of the push-buffer. These fix-ups can be written directly to the push-buffer memory.

Per-call fix-ups update push-buffer data that has to be changed on a per-call basis, such as transform data. Because the same push-buffer may be enqueued multiple times for use by the GPU with different per-call fix-ups for each, per-call fix-ups cannot be written to the push-buffer at the time that the push-buffer is enqueued, but rather must be done at the time that the GPU dequeues the push-buffer. Because the fix-up data must persist until the GPU dequeues the push-buffer, per-call fix-ups are specified to the IDirect3DDevice8::RunPushBuffer function using a "fix-up object," which is a Direct3D resource that inherits all of the time management facilities associated with Direct3D resources (such as IDirect3DResource8::IsBusy).

Push-buffer fix-up functions are provided to create both types of fix-ups. The only invocation difference is that per-call fix-ups specify a fix-up object to IDirect3DPushBuffer8::BeginFixup, which is automatically appended to when the fix-up functions are called. It's generally recommended that callers employ ping-pong fix-up objects, so that when one fix-up object is full, the other can be appended to without having to flush the whole graphics pipeline.

Fix-ups are identified by their offset location in the push-buffer. While in a IDirect3DDevice8::BeginPushBuffer/IDirect3DDevice8::EndPushBuffer bracket to record a push-buffer, IDirect3DDevice8::GetPushBufferOffset may be called before any Direct3D function call to determine the offset of that Direct3D function call in the recorded push-buffer.

Only a limited subset of Direct3D function calls may be fixed-up, and for each such Direct3D function call there is a corresponding fix-up function. The complete list of the Direct3D functions (that have fix-ups) and their corresponding fix-up function is below.

Note that it is not possible to fix-up IDirect3DDevice8::SetStreamSource or IDirect3DDevice8::SetTransform calls. This is because those calls are not self-contained enough to provide all dependent data necessary for Direct3D to program the hardware immediately, and so processing of those function calls is always deferred until the next render call. Consequently, IDirect3DDevice8::SetVertexShaderInput and IDirect3DDevice8::SetModelView/IDirect3DDevice8::SetVertexBlendModelView must be used instead (even in normal use these functions are faster than their equivalents).

The Xbox GPU has no notion of a hardware transform stack, so all transforms must be explicitly fixed-up.

When creating a fix-up object, all the fix-ups must be created in order of increasing offsets.

Fix-ups must have the same push-buffer "signature" as was recorded into the push-buffer. This means that it's not possible to change the type of objects. For example, if IDirect3DDevice8::SetTexture was called with a linear texture while recording, the IDirect3DPushBuffer8::SetTexture fix-up cannot be called with a swizzled texture. (The only exception to this is that IDirect3DDevice8::SwitchTexture has the same signature as IDirect3DDevice8::SetTexture, and so IDirect3DPushBuffer8::SetTexture may be used to fix-up IDirect3DDevice8::SwitchTexture calls.)

The push-buffer "signature" of functions is not guaranteed to be the same across different builds of Direct3D. This means, for example, that a push-buffer should not be recorded with one build version and fix-ups applied with a different build version. It's best to regenerate static push-buffers from build-to-build if they contain data that needs to be fixed-up.

The IDirect3DPushBuffer8::Verify method is provided as a debug aid for verifying that all resource references are fixed up before the push-buffer is executed (to avoid cases in which a fix-up is missed and the hardware continues to use a stale, invalid address.)
