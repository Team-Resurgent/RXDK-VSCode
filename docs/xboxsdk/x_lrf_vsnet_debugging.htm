Debugging with Visual Studio .NET

Visual Studio .NET offers a much richer set of debugging options than were available in the Visual C++ 6.0 environment.

Xbox-Specific Debugger Changes and Improvements

  • Edit and Continue is now supported for Xbox. You can make changes to your source code while debugging and have the changes take effect immediately, without having to restart your Xbox.

    Note    If you imported your Xbox project from Visual C++ 6.0, you need to enable Edit and Continue support in the project before it is available. See the Enabling Edit and Continue Support section for more details.

  • Editor window tool tips and the Watch window support Direct3D data structures such as D3DVECTOR. While debugging, you can see some of the values in these structures either by hovering the mouse over a variable name in the editor or by looking at the appropriate variable description in the Watch window.
  • The Visual Studio .NET debugger has been extended to recognize and debug Xbox crash dumps. Crash dumps are complete memory images that can be used to debug Xbox applications after the fact. See the Debugging Crash Dumps section for more details.
  • In Visual Studio .NET, it is not necessary for the EXE file to be present on the development system when debugging to an Xbox console.
  • Because XbWatson is itself a debugger, it interferes with debugging under Visual Studio .NET. Do not run the XbWatson tool while debugging in Visual Studio .NET.

General Debugger Changes and Improvements

  • You can set or remove breakpoints by simply clicking in the breakpoint gutter in the left margin of a source file.
  • The debugger supports up to four memory windows simultaneously.
  • Many of the options previously located in dialog boxes are now located in docking windows. To open the debugger windows, click the Windows item on the Debug menu. (This is a bit strange because all of the other windows in the IDE are located on the View menu.)
  • The Registers window allows access to all of the CPU registers. Right-click the Registers window to retrieve a menu of registers, and then select the registers that you want to view.
  • The Threads window is modeless, making it easier to debug multithreaded applications.
  • The Modules window can be used to reload symbols, making it easier to debug problems with your symbol path. From the first view, right click the module for which you want to reload symbols, and then click Reload. (In Visual Studio 6.0, you had to restart the debugger when you changed your symbol paths.)
  • Visual Studio .NET supports loading symbols from a symbol server. This is especially useful for larger organizations. You can use a symbol server to share symbol files for the kernel and for your Xbox games across all your Xbox consoles. Loading symbols from a symbol server is surprisingly easy to set up (see the Microsoft Windows Driver Development Kit for more information.)
  • The debugger can attach to a running process on an Xbox console. This feature is controlled through the Attach to Process dialog box, which you can access from the Debug Process item on the Tools menu. To attach to a running process on the Xbox console, open the Attach to Process dialog box, and then click Xbox Console Transport.
  • You can customize the way that the debugger's "data tips" display structures. The Autoexp.dat file found in the C:\Program Files\Microsoft Visual Studio .NET\Common7\Packages\Debugger directory allows you to customize "data tips" pop-up debug information. This file also allows you to customize how data is displayed in the watch windows.
  • The debugger supports better debugging of template classes. This is because the debug information format was expanded to support longer mangled names.
  • When opening an EXE or XBE to debug in Visual C++ 6.0, you were able to use File.OpenFile nameofexe. However, in Visual Studio .NET, using File.OpenFile nameofexe opens the EXE or XBE in the binary editor or resource editor. To open an EXE or XBE for debugging in Visual Studio .NET, you must click File, Open, Project, and then click the name of the EXE to load.
  • In Visual Studio .NET, the directories used to search for symbols and source files have changed slightly. To find the list of directories that the debugger will use, first right-click the solution node, and then click Properties on the context menu. Next, to specify where the debugger will search, click either the Debug Source Files page or the Debug Symbol Files page. In Visual C++ 6.0, the directories to search for files were located in the Tools.Options.Directories dialog box. It is important to note that, although a similar page exists in Visual C++ 7.0, the debugger ignores the settings on that page.
  • The Visual Studio .NET debugger was expanded to support simultaneous debugging of multiple Win32 processes. However, this feature does not work with Xbox consoles. To simultaneously debug multiple Xbox consoles, you must run multiple copies of the VS.NET IDE. (You must run one copy of VS.NET IDE for each Xbox console you wish to debug.)

Known Issues

This section lists known issues with using Visual Studio.NET to debug Xbox projects.

  • After hitting an assert or breakpoint, the debugger in Visual Studio .NET sometimes displays incorrect values for local variables in functions on the callstack other than the current function. This problem is caused by local variables that use __declspec(align()). Xbox developers most commonly encounter this issue when using the XGMATRIX type. There is no workaround for this issue, which affects both the Visual Studio .NET and VC6 debuggers. This issue will be corrected in an upcoming release of Visual Studio .NET.

  • If you use a watch expression in Visual Studio .NET that requires function evaluation, and you use Visual Studio .NET rather than XbWatson to save minidumps, then the stack flags in the minidump will be incorrect.

  • A minidump generated while running an XBE that does not contain debug information may fail to load symbols even if the matching PDB file is present when the minidump is opened for debugging in Visual Studio .NET. This is not a problem if the /DEBUG switch was used when the XBE was built.

Debugging Crash Dumps

A crash dump is a 128MB file containing the complete contents of the Xbox console's RAM at the time of an exception. Both Xbox Development Kits and Debug consoles can generate crash dumps, provided the consoles have been recovered with the August 2002 or later recovery disc image (or remote recovery).

The Xbox debugger run time (XBDM) generates a crash dump when the system reaches specific stopping events and it is in a state that prevents it from being debugged via an Ethernet connection. The most common reason for being in such a state is that the processor is running at a raised Interrupt Request Level (IRQL), such as DPC. In this state, the stopping events that generate a crash dump include unhandled exceptions (which can be thrown by the processor, XTL code, or your source code), assertions, RIPs, bugchecks, embedded breakpoints, and data breakpoints. Breakpoints set via the Ethernet port using Visual Studio .NET are not data breakpoints. Therefore, they do not generate crash dumps. Instead, at the next opportunity, XBDM sends debug information indicating that the event was missed. If a serial debugger, such as i386kd or WinDBG, is connected to the development console at the time of the event, XBDM does not generate a crashdump. Rather, it routes the event to the serial debugger.

Because a crash dump occurs at a system-terminating event, the connected debugger has no way to know that a crash dump is in progress. This can cause the Xbox console and the debugger to appear hung. When the Xbox console is creating a crash dump, it stops responding to user input, and its power LED flashes back and forth between red and yellow lights. It takes the Xbox console up to two minutes to store a complete memory image to disk, at which point the Xbox console reboots. While the console is rebooting after a crash dump, the LED flashes between green and yellow. Once the console has finished booting, the LED returns to its usual solid green light.

After the Xbox console has rebooted, the crash dump is available in the root of the xe: drive on the console. XBDM always writes the crash dump to a file named CrashDump.xdmp. To work with the crash dump, you need to copy it to your development machine. This is usually done with the XbCp tool. You should also rename the copy of the file on the development PC so it has an .xdmp extension, because Visual Studio .NET recognizes this extension as a crash dump. Renaming the file enables you to double-click it to open the crash dump in the IDE. To copy the crash dump to your development system and rename it in one operation, use the following command line: 

xbcp xe:\crashdump.xdmp c:\myfolder\mycrashdump.xdmp

Once the crash dump is on your development system, you can open it for debugging in Visual Studio .NET by double-clicking the .xdmp file in Windows Explorer. Alternately, you may open it by clicking Open.Project on Visual Studio's File menu. To debug the crash dump once it is open, press F5.

Note    If you have one of the early green debug kits, you may encounter problems when working with crash dumps if you also have an early version of the XDK. To avoid these problems, ensure that you have the latest version of the XDK.

The XTL includes a function called DmCrashDump that programs can use to force an Xbox development console to generate a crash dump at any time. This function is available via xbdm.dll on the Xbox side. It can also be called through xboxdbg.dll on the PC side.
