Competition Configuration

Before you can offer a competition on Xbox Live, you must define a configuration for it. Configurations contain one or more templates. Each template defines the characteristics of a competition. You create configurations for competitions using the XLAST tool provided in the XDK.

You begin the process of creating a competition by starting the XLAST New Competitions Project Wizard. The wizard creates a configuration project to which you can add competition templates. Each template defines one type of competition. You can create as many competitions from a template as you need.

Define the characteristics of a competition by adding attributes to its template. You can also add constants to the template that define the values that can be assigned to the attributes.

The Xbox Live service uses the templates from your configuration package to create a database containing information about each competition. As the competition progresses, your game updates the contents of the database using the competitions functions. These functions perform their tasks by sending queries to the competition's database. You can use the XLAST tool to define your own queries that your title can send to the competition's database.

Note    In the competitions functions, the dwTemplateID parameter is used to reference the appropriate competition template.

The database for your competitions is also called a dataset. The dataset's information is kept in the following three tables.

Competitions table
The competitions currently running, open for registration, or scheduled for cleanup. The list is limited to the set of all competitions based on the selected template.
Entrants list
The players or teams enrolled in each competition.
Events list
All events in each competition. This include events already played, currently in progress, and scheduled for the future.

These three tables are referred to as the "targets" of the queries that update the dataset. These queries are configured via the Queries node in the XLAST Competition Configuration tool.

Note    Dataset schemas and indices cannot be modified in production, that is, after they have been released to Xbox Live. The procedure required for production is to submit a new dataset and autoupdate for the client to use the new dataset. This new dataset replaces the one in production. Although the dataset and autoupdate can be in PartnerNet, this results in the loss of existing competitions in those datasets. Constants and queries can be updated for a dataset in production.

To provide maximum flexibility for title developers, the competitions and events targets can be configured to store title-specific custom attributes. You define these attributes in XLAST by including them in the various queries associated with the competition template.

It is also possible to add title-specific custom attributes to the entrants target. However, you cannot do this through XLAST. You must add them to XLAST's XQS output. The following example adds the competition name to the entrants dataset.

<dataset id="2" quota="-1" name="elimination entrants" typeid="0">
    <schema>
        <!-- note: entity_id, creator_puid, etc that are "globals" and part of every dataset.
        <attribute name="entity_id"    hexid="0x80000000" kind="int" />
        <attribute name="owner_id"    hexid="0x80000001" kind="int" />
        -->
        <!-- Standard competition attributes -->
        <attribute name="att_comp_id" hexid="0x80010000" kind="int" default="0"/>
        <attribute name="att_comp_status" hexid="0x8001001D" kind="int" default="0" />
        <attribute name="att_puid" hexid="0x80020000" kind="int" default="0" />
        <attribute name="att_status" hexid="0x80020001" kind="int" default="0" />
        <attribute name="att_current_round" hexid="0x80020003" kind="int" default="0" />
        <attribute name="att_current_event" hexid="0x80020007" kind="int" default="0" />
        <attribute name="att_current_start" hexid="0x80020008" kind="int" default="0" />
        <attribute name="att_sort" hexid="0x80020009" kind="int" default="0" />
        <attribute name="att_private_slot" hexid="0x80020004" kind="int" default="0" />
        <attribute name="att_seed" hexid="0x80020005" kind="int" default="0" />
        <attribute name="att_comp_name" hexid="0x8011000E" kind="string" maxlen="32" default="" />
    </schema>
    ...
</dataset>

The custom attribute in the preceding example is called att_comp_name. If you used this attribute in your game, your queries would need to be manually modified to return the attribute value. The "My Competitions" query, for instance, would resemble the following.


<query hexid="0x1" op="search" name="My Competitions">
    <params>
        <param name="p_puid" hexid="0x82020000" kind="int" />
        <param name="p_comp_status" hexid="0x8201001D" kind="int" />
    </params>
    <filters>
        <filter left="0x80020000" op="==" right="0x82020000" />
        <filter left="0x8001001D" op="==" right="0x8201001D" />
    </filters>
    <sortops>
        <sortop hexid="0x80020009" dir="ascending" />
    </sortops>
    <returns>
        <return name="att_comp_id" hexid="0x80010000" />
        <return name="att_comp_status" hexid="0x8001001D" />
        <return name="att_status" hexid="0x80020001" />
        <return name="att_current_round" hexid="0x80020003" />
        <return name="att_current_event" hexid="0x80020007" />
        <return name="att_current_start" hexid="0x80020008" />
        <return name="att_sort" hexid="0x80020009" />
        <return name="att_comp_name" hexid="0x8011000E"  />
    </returns>
</query>

Note    If you regenerate your queries with XLAST, XLAST overwrites the custom attributes for the entrants dataset. You must manually edit them to reinsert the custom attributes.

Once the attributes have been added to the schema for the entrants dataset, your game must set them using XOnlineCompetitionManageEntrant to execute to join (XONLINE_COMP_ACTION_JOIN) the player into a competition. At this time, the title retrieves any attributes based on other tables (for example, competition dataset attributes such as competition name, start time, and so forth) and then set those for the player.

When duplicating attributes from other datasets, the custom attributes in the entrants dataset are not joined between datasets. If the values change in another dataset, the changes are reflected in the entrants dataset. Therefore, the technique show here for inserting custom attributes is mainly for static attributes that are typically initialized once and then not changed for the lifetime of the competition.
