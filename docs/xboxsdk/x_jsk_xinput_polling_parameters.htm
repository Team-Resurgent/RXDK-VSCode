XINPUT_POLLING_PARAMETERS

The XINPUT_POLLING_PARAMETERS structure contains arguments that enable fine control of the polling method XInputOpen.

typedef struct _XINPUT_POLLING_PARAMETERS
{
    BYTE       fAutoPoll:1;
    BYTE       fInterruptOut:1;
    BYTE       ReservedMBZ1:6;
    BYTE       bInputInterval;  
    BYTE       bOutputInterval;
    BYTE       ReservedMBZ2;
} XINPUT_POLLING_PARAMETERS, *PXINPUT_POLLING_PARAMETERS;

Members

fAutoPoll
Chooses automatic or manual polling of input devices. If FALSE, the title must call XInputPoll to initiate poll requests before calling XInputGetState. The default is TRUE.
fInterruptOut
Determines whether output uses reserved bandwidth. If TRUE, a device reserves a time slot to send a packet every bOutputInterval milliseconds. Thus the maximum output latency is bOutputInterval. If fInterruptOut is FALSE, no bandwidth is reserved, and data is sent as soon as possible. In most cases, setting fInterruptOut to FALSE results in lower latency, but there is no guarantee. The default is FALSE.
ReservedMBZ1
Reserved for future use. This should be initialized to 0.
bInputInterval
Determines the hardware polling interval for input in milliseconds. The value will be rounded down to the nearest power of two. The default is 8 milliseconds for gamepads, and 16 milliseconds for the debug keyboard.
bOutputInterval
Determines the hardware polling interval for output. The value will be rounded down to the nearest power of two. This field applies only if fInterruptOut is TRUE. The default is 0 milliseconds (a value similar to bInputInterval is recommended when setting fInterruptOut to TRUE).
ReservedMBZ2
Reserved for future use. This should be initialized to 0.

Remarks

XINPUT_POLLING_PARAMETERS exposes advanced features. Some understanding of the hardware is required for effective use. This discussion considers input and output separately.

Input is received via a hardware polling mechanism. Polling is initiated through a software request. At each polling interval, the hardware has an opportunity to respond with new data. Xbox devices will send new data only when their state changes. (A different mechanism is used to get the initial state of the device when the device is opened.) When a device sends data, an interrupt is generated and the driver updates the device state. XInputGetState always retrieves the cached state of the device. After receiving data, the hardware ceases to poll until an additional software request is submitted. When a device is opened with fAutoPoll set to TRUE, the driver automatically submits a new request every time data is received. When fAutoPoll is FALSE, the title must submit the request by calling XInputPoll. In either case, bInputInterval determines the fixed hardware polling interval in milliseconds.

The clock used to time the polling interval is independent of the video frame clock. To avoid near resonance effects, the default polling rate is set at 8 milliseconds for game controllers. A slightly more efficient method is to synchronize data acquisition with the video frame clock. Assuming a title's render loop is synchronized with the video blanking interval, a game could call XInputPoll at a fixed point in the render loop at least bInputInterval plus a millisecond prior to calling XInputGetState. bInputInterval would then be set to a small value, such as 2 milliseconds. The net effect is that data acquisition is synchronized to occur from 1 to 3 milliseconds before the call to XInputGetState, which occurs at a fixed point in the render loop. Furthermore, the maximum number of interrupts is reduced by about half. It should be noted that in this scenario, the device will not respond if the data has not changed; the request will remain outstanding. This causes no harm, as the driver ensures that only one request is pending at a time.

Output is slightly different. If fInterruptOut is TRUE, a similar polling schedule is used, but there is nothing to poll. When the title submits data with XInputSetState, the hardware waits until the next appointed interval, determined by bOutputInterval, before sending it. The advantage is that the device has reserved that interval and can thus guarantee maximum latency. The disadvantages include: an additional 60 bytes or so of memory are allocated during XInputOpen, the data cannot be sent until that interval occurs(even if the bus is not busy), and in the future reserved bandwidth may become a limited resource. With the initial mix of available devices, it is expected that setting fInterruptOut to FALSE will achieve better performance.

If NULL is passed as the pPollingParameters argument to XInputOpen, the default values will be selected.
