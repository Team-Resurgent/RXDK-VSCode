XMemFree

Application-defined memory deallocation callback function that replaces the memory deallocator used by the XTL libraries.

VOID WINAPI XMemFree(
    PVOID pAddress,
    DWORD dwAllocAttributes
);

Parameters

pAddress
[in] Specifies the base address of the memory to be freed. This parameter must be the address returned by the XMemAlloc function when the memory was allocated.
dwAllocAttributes
[in] An XALLOC_ATTRIBUTES value that specifies the allocation attributes of the memory to be freed. These attributes should match those that were specified when the memory was allocated (but see Remarks below).

Return Values

This function does not return a value.

Remarks

If a title implements an XMemFree function, all deallocations performed by the XTL libraries will instead call the title's XMemFree routine. The title must deallocate the memory as requested. The title can simply use the XPhysicalFree and HeapFree functions to provide the underlying heap support, or call the default XMemFreeDefault if desired. Alternatively, the title can implement its own heap routines and thus replace the default heap deallocator.

XTL components that need to deallocate memory (graphics, sound, networking, and so on) all call XMemFree instead of other deallocation routines. The default XMemFree then calls XPhysicalFree or HeapFree as appropriate. A title can provide its own XMemFree function, thus overriding the default, and allowing the title to trace XTL deallocations and to replace the heap deallocator. The title should provide all of the allocator replacement functions (XMemAlloc, XMemFree, and XMemSize) or none at all. The allocator functions to use (either the default or title-defined versions) are determined at link time, so the title imposes no additional overhead by supplying its own versions.

The title can, if desired, call its own implementation of XMemFree for deallocations as well, thus routing all title and XTL deallocations through a single deallocation routine.

If the heap is storing allocation attributes along with the allocations, XMemFree can verify that the attributes specified in dwAllocAttributes match those stored with the allocation (in order to detect allocation/deallocation mismatches). The dwHeapTracksAttributes member of the structure specified by dwAllocAttributes indicates whether the heap is responsible for storing allocation attributes.

Not all XTL libraries pass XALLOC_ATTRIBUTES flags in dwAllocAttributes that precisely match those used for the allocation. For example, D3D may allocate physical memory with XALLOC_MEMPROTECT_WRITECOMBINE, XALLOC_MEMPROTECT_NOCACHE, or XALLOC_MEMPROTECT_READWRITE, but always specifies XALLOC_MEMPROTECT_READWRITE when freeing that memory. For performance reasons, D3D does not track attributes for physical memory allocations. Debug asserts caused by this attribute mismatch can be safely ignored.
