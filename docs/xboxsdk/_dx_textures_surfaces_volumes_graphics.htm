Textures, Surfaces, and Volumes

The resource header structures for IDirect3DCubeTexture8, IDirect3DSurface8, IDirect3DTexture8, IDirect3DVolume8, and IDirect3DVolumeTexture8 all share a similar structure definition.

If there is a need to manually create a resource, it is recommended that the title use XGSetCubeTextureHeader, XGSetSurfaceHeader, XGSetTextureHeader, or XGSetVolumeTextureHeader.

struct _ResourceName {
  // Resource specific declarations
  // ...
  
  DWORD Common;
  DWORD Data;
  DWORD Lock;
  
  DWORD Format;
  DWORD Size;
};

Members

Common
Holds all of the information common to all resources inheriting from IDirect3DResource8. Information is stored in the following bit fields.
Field Bits Description
REFCOUNT 16 Reference count for the resource, which will be destroyed if this value ever goes to zero.
TYPE 3 Holds the type of the resource. For all textures, this will be D3DCOMMON_TYPE_TEXTURE. For any surface or volume, this will be D3DCOMMON_TYPE_SURFACE.
INTREFCOUNT 4 Internal reference count that is used to determine whether IDirect3DDevice8 is using this resource.
VIDEOMEMORY 1 This bit is not used.
D3DCREATED 1 Flag to indicate that the resource was created using one of the Create methods of IDirect3DDevice8. This flag should not be set for manually created resources.

Surfaces have one additional flag that may get set: D3DSURFACE_OWNSMEMORY. If this flag is set, it indicates that the surface owns the memory pointed to by its Data field and that it should free that memory when its reference count goes to zero. The memory will be freed with D3D_FreeContiguousMemory.

Data
The offset of the data for this resource in physical memory. The memory for the data of a resource must be stored in contiguous, write-combined memory. The data allocation for each resource must be aligned on a specific boundary depending on the resource type.

All textures must be aligned using D3DTEXTURE_ALIGNMENT. Each face of a cube map must be aligned on a new D3DTEXTURE_CUBEFACE_ALIGNMENT byte boundary.

All surfaces must be aligned using D3DSURFACE_ALIGNMENT.

The contiguous memory allocation ensures that the consecutive virtual pages of the allocation will use consecutive physical pages, because the graphics processor does not have any support for virtual memory. The memory must be write combined to ensure coherency between the graphics processor and the CPU. The write-combined mapping inhibits the CPU's normal caching behavior but will combine writes in short-term buffers to improve write speeds. Direct3D automatically flushes these buffers whenever any new commands are sent to the graphics processor.

The data for a texture consists of the raw texels for the texture; there is no header information in the data. All of the pieces of a complicated texture (mipmaps, cube map) must be located within a single contiguous allocation that must be aligned on a 128-byte boundary.

Textures with multiple mipmap levels store the levels ordered from largest to smallest with no padding between each level. Cube-mapped textures have the full mipmap of each face aligned at the next 128-byte boundary in D3DCUBEMAP_FACES order.

So a 4×4×1 three-mipmap-level cube-mapped texture with four bytes per texel would be laid out as follows:

Offset Description
0 Data for first level (4×4×1), positive X face
64 Data for second level (2×2×1), positive X face
80 Data for third level (1×1×1), positive X face
128 Data for first level (4×4×1), negative X face
192 Data for second level (2×2×1), negative X face
208 Data for third level (1×1×1), negative X face
256 And so on

The data for a surface/volume consists of the raw pixels for the surface/volume with no header information.

Lock
Contains the fence that must be crossed before this resource is no longer used by the graphics processor. It will contain zero if there is no possibility of this resource being in use or if it is currently set on the device. This field should always start out as zero and should never be modified by the title.
Format
Holds the description (format) of the texture or surface.

The formats of a texture or surface can be grouped into four categories:

  1. Swizzled—The swizzled formats indicate that the texels in the surface/texture have been reordered for improved locality of reference during the rendering process. Rendering will be fastest when using textures of this format, but these are not the most efficient formats for surfaces being used as a render target. Reading or writing the data of one of these textures or surfaces should be done through the swizzle helper routines defined in xgraphics.h.
  2. Compressed—The DXT compressed formats.
  3. Linear—The linear formats organize the texels in a way that is compatible with DirectX 8.0. Rendering from a linear texture will be slower than rendering from a swizzled texture. The YUV formats are considered linear formats.
  4. Depth—The fixed and floating-point formats that may be used as a depth buffer. The depth formats are also either linear or swizzled and must follow all of those restrictions as well. The graphics processor does not support rendering to a swizzled depth buffer.

Textures and surfaces have the following restrictions:

  • Swizzled and compressed formats require that the dimensions of the texture/surface be a power of two.
  • Linear formats do not support mipmaps (mipmap level must be 1).
  • Linear and depth formats do not support cube maps.
  • Linear and depth formats do not support volumes (dimension must be 2).
  • 2D textures/surfaces can have a maximum width/height of 4,096 texels.
  • 3D textures/surfaces can have a maximum width/height/depth of 512 texels.
  • Cube maps must be square.
  • 3D textures cannot be cube mapped.

Information is stored in the following bit fields:

Field Bits Description
DMACHANNEL 2 Must be D3DFORMAT_DMACHANNEL_A.
CUBEMAP 1 Flag that indicates that this texture is a cube map.
BORDERSOURCE_COLOR 1 When this flag is set, the color specified in the texture stage D3DTSS_BORDERCOLOR will be used as the border for the texture.
DIMENSION 4 The number of dimesions of the texture or surface. This may be either two or three.
FORMAT 8 The D3DFORMAT of the surface or texture.
MIPMAP 4 The number of mipmap levels in this texture. For a surface, this is always one.
USIZE 4 The log2 of the u-dimension of the texture or surface. This will be used only if the format of the texture or surface is either swizzled or compressed.
VSIZE 4 The log2 of the v-dimension of the texture or surface. This will be used only if the format of the texture or surface is either swizzled or compressed.
PSIZE 4 The log2 of the p-dimension of the texture or surface. This will be used only if the format of the texture or surface is either swizzled or compressed.
Size
Holds the size and pitch of linear textures. These textures do not have to have dimensions whose size is a power of two. This member will be zero for swizzled and compressed textures and will always be nonzero otherwise. Information is stored in the following bit fields:
Field Bits Description
WIDTH 12 Width-1 of the texture/surface in texels/pixels.
HEIGHT 12 Height-1 of the texture/surface in texels/pixels.
PITCHADJ 8 Contains the actual pitch value divided by 64 and minus 1.
