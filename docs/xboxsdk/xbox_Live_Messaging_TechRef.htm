Message Components

The basic building block of a message is the message property. These properties are the smallest units of data on a message. Each property is given a unique ID and a type that defines the property's contents. The type and ID together form a property tag that is used when setting and retrieving the property from a message. Titles are required to request only the properties they are interested in rather than being able to see all the properties on a message. This is done so that a particular message that a title is familiar with handling does not break the title if new properties are added to the message type in the future. Titles may embed title-specific properties for use within the title as they see fit. Messages contain three property types.

  • Summary Information
  • Message Details
  • Attachments

Summary Information

The message summary contains the common set of properties a developer can expect to find on any message. Think of this as the base metadata about the message. This information is used to make initial decisions about the routing and displaying of the message to the user. The size of the message summary is 64 bytes. During the logon process, the summary information for each message is retrieved, remains resident on the client, and is kept up-to-date via notifications.

Message Details

Message details are a collection of properties applicable to the particular message type. Some messages have required properties that are predefined and expected in the message details. Developers may define their own properties to embed in the details. Details are a maximum of 4,096 bytes in size. The details of a message are retrieved when the user opens the message.

Attachments

A message may have attachments that accompany the message. Imagine a user sharing a map created in-game with a friend and annotating the map with a voice message. The map would be attached to the message. Attachments are represented as URLs to the Xbox Live storage service and stored as properties in the message details. Attachments are retrieved upon user action after the details information has been retrieved. Attachments are governed by the policy and requirements of the Xbox Live storage service.

Message Structure

The following structure holds the message summary.

typedef struct _XONLINE_MSG_SUMMARY
	{
	    XUID        xuidSender;        // User ID of sender
	    BYTE        bMsgType;          // Type of the message
	    ULONGLONG   qwMessageContext;  // Message-specific context value
	    FILETIME    ftSentTime;        // Time at which message was sent, in Coordinated Universal Time (UTC) format
	    DWORD       dwMessageID;       // ID of message
	    DWORD       dwMessageFlags;    // Flags describing message
	    DWORD       dwSenderTitleID;   // ID of title in which message was sent
	    WORD        wExpireMinutes;    // Offset in minutes from the sent time
	    WORD        cbDetails;         // Size of details BLOB, excluding downloadable content
	    char        szSenderName[XONLINE_GAMERTAG_SIZE]; // Gamertag of sender
	} XONLINE_MSG_SUMMARY, *PXONLINE_MSG_SUMMARY;

ULONGLONG qwMessageContext

The message context (qwMessageContext) is used to pass type-specific information about the message, the main purpose of which is to immediately give the title enough knowledge about the message to make an informed decision on how to visually indicate pending messages to the user. Think of this as "bread crumbs" that help lead the user to the appropriate place to take action on the message. Including this information in the summary alleviates the need to make a separate asynchronous request to get message details. The sender context is well-defined for each known message and may be used by title-specific messages in a manner applicable to that message's application.

Friend Request

Description Another player is requesting a friend relationship, either through the player's menu or via the Dashboard.
Message type XONLINE_MSG_TYPE_FRIEND_REQUEST
Sender context <none>
Required properties <none>
Delivered cross-title? Yes
In-game notification required? Yes.
Expiration policy 45 days

Game Invite

Description A game invite that was sent from the friends, teams, or competitions UI
Message type XONLINE_MSG_TYPE_GAME_INVITE
Sender context Sender contextual, could be <none>, Team_ID, or CompID
Required properties XONLINE_MSG_PROP_SESSION_ID (ID of the session the user is being invited to)
Delivered cross-title? Yes, if recipient is already a Friend
In-game notification required? Yes.
Expiration policy Transient

Team Recruitment

Description A message sent to solicit a player to join a team
Message type XONLINE_MSG_TYPE_TEAM_RECRUIT
Sender context Team_ID
Required properties XONLINE_MSG_PROP_TEAM_NAME (string indicating name of team)
Delivered cross-title? No
In-game notification required? Optional
Expiration policy Title-specified, max 45 days

Competition Recruitment

Description A message sent to solicit a player to join a competition event
Message type XONLINE_MSG_TYPE_COMP_RECRUIT
Sender context Comp_ID
Required properties

XONLINE_MSG_PROP_COMP_NAME (string indicating name of competition)

XONLINE_MSG_PROP_COMP_EVENT_START (starting date of the event)

XONLINE_MSG_PROP_COMP_ADMIN (XUID of the admin)

XONLINE_MSG_PROP_COMP_REG_CLOSE (when the competition is available for registration)

XONLINE_MSG_PROP_COMP_RECURRENCE
Delivered cross-title? No
In-game notification required? Optional
Expiration policy Title-specified, max 45 days

Competition Match Reminder

Description As a precursor to an upcoming match, this message is sent to a player who previously joined the competition event. This message is created by the service and delivered to a player shortly before the match begins.
Message type XONLINE_MSG_TYPE_COMP_REMINDER
Sender context Comp_ID
Required properties

XONLINE_MSG_PROP_COMP_NAME (string indicating name of competition)

XONLINE_MSG_PROP_COMP_EVENT_START (starting date of the event)
Delivered cross-title? No
In-game notification required? Optional
Expiration policy 15 minutes from start of competition

Custom Title Message

Description Titles are each allowed to create a customized title-specific message for use within the title as they see fit. A title may have multiple message types, but they will all appear as this type to the messaging service and to external clients. A title can use the title-reserved bits in the message flags or the sender context to distinguish its own internal types. Custom messages can have any combination of voice, text, or data attachments.
Message type XONLINE_MSG_TYPE_TITLE_CUSTOM
Sender context <Title's discretion>
Required properties <Title's discretion> (A title may define any properties that are needed for its custom type.)
Delivered cross-title? No
In-game notification required? <Title's discretion>
Expiration policy Title-specified, max 45 days

Personal Voice Message

Description A personal voice message sent between parties with an existing friend, team, or competition relationship
Message type XONLINE_MSG_TYPE_VOICE
Sender context <none>
Required properties

XONLINE_MSG_PROP_VOICE_DATA (Storage URL to the actual voice data)

XONLINE_PROP_VOICE_DATA_CODEC (codec and bitrate used to encode the voice data)
Delivered cross-title? No
In-game notification required? Optional, with envelope
Expiration policy 7 days

Personal Text Message

Description A personal text message sent between parties with an existing friend, team, or competition relationship
Message type XONLINE_MSG_TYPE_TEXT
Sender context <none>
Required properties

XONLINE_MSG_PROP_TEXT (text of the message)

XONLINE_PROP_TEXT_LANGUAGE (language of the text in the XMSG_PROP_TEXT property)
Delivered cross-title? No
In-game notification required? Optional, with envelope
Expiration policy 7 days

Publisher-Specific Message (MOTD)

Description Publishers are each allowed to create a publisher-specific message targeted to an individual title. This message will be delivered to all users of that title.
Message type XONLINE_MSG_TYPE_TITLE_CUSTOM
Sender context <none>
Required properties

XMSG_PROP_TEXT (text of the message)

XMSG_PROP_TEXT_LANGUAGE (language of the text in the XMSG_PROP_TEXT property)
Delivered cross-title? No
In-game notification required? <Title's discretion>
Expiration policy Title-specified, max 45 days. This is a special case whereby the publisher may choose an absolute expiration date or a delta consisting of the number of days from the time the message is first delivered to the user.

Message Flags

The message flags are part of the summary properties. Titles may consult these bits to make informed decisions on what to display in-game or when constructing a message list.

Flag Value Description
XONLINE_MSG_FLAG_REQUIRED 0x00000001 The user is required to read this message.
XONLINE_MSG_FLAG_RECOMMENDED 0x00000002 The user has a system recommended message.
XONLINE_MSG_FLAG_HAS_VOICE 0x00000004 This message contains a voice attachment.
XONLINE_MSG_FLAG_HAS_TEXT 0x00000008 This message contains a text body
XONLINE_MSG_FLAG_READ 0x00000010 This message has been read.
XONLINE_MSG_FLAG_NON_EXPORTABLE 0x00000020 This message should only be displayed on Xbox consoles, not the Web.
XONLINE_MSG_FLAG_TEAM_CONTEXT 0x00000040 This message's sender context refers to a team ID.
XONLINE_MSG_FLAG_COMP_CONTEXT 0x00000080 This message's sender context refers to a competition event or entity ID.
XONLINE_MSG_FLAG_ALTERNATE_TITLE 0x00000100 This message is from an alternate title of the currently logged on title (only available on Xbox consoles).
XONLINE_MSG_FLAG_MARKETING 0x00000200 Used on title-sent messages to indicate the message is marketing-related and opt-in settings should be checked.
XONLINE_MSG_FLAGS_TITLE_RESERVED 0xFF000000 Flags reserved for title custom messages.

Global Property Flags

Some property flags are global and are allowed in any message type. All message types can contain a voice attachment specified in the following properties:

  • XONLINE_MSG_PROP_VOICE_DATA
  • XONLINE_MSG_PROP_VOICE_DATA_CODEC
  • XONLINE_MSG_PROP_VOICE_DATA_DURATION

All three of the above properties should be specified together. The XONLINE_MSG_FLAG_HAS_VOICE message flag can be used to test for the presence of these properties before the message details have been retrieved.

Additionally, all message types can contain a text summary specified in the following properties:

  • XONLINE_MSG_PROP_TEXT
  • XONLINE_MSG_PROP_TEXT_LANGUAGE

Both of the above properties should be specified together. The XONLINE_MSG_FLAG_HAS_TEXT message flag can be used to test for the presence of these properties before the message details have been retrieved. The text summary should be 256 characters or less and may be used by Xbox Live anywhere that a summary of the message is useful, such as in an alert, on the Web, or in the Xbox Dashboard.

Transient and Persistent Messages

The messaging service sends both transient and persistent messages to players. Transient messages are time critical. Invitations to join a game fall into the category of transient messages. The messaging service keeps a maximum of 100 persistent messages and 25 transient messages in each player's queue.
