Kernel Memory Management

The kernel provides the basic memory management functionality for the Xbox. The Xbox kernel implements virtual memory, but not demand paging.

Virtual Memory

With virtual memory, there is a translation layer between memory addresses used by the game title and the physical RAM that those addresses represent. Titles can address up to 4 GB of virtual memory using 32-bit linear addresses. The 64 MB of available physical RAM is mapped into that 4 GB address space in 4 KB chunks called pages. The kernel's virtual memory management support is exposed to titles through the virtual memory functions in the XTL, such as VirtualAlloc. Most virtual addresses used by titles will be in the lower two gigabytes, but some virtual addresses, such as thread stacks and physical memory allocations (discussed below) will be in the upper two gigabytes of the address space.

The advantage to using virtual memory addresses backed by pages of RAM is that available RAM can be mapped into a contiguous virtual address range even if the RAM itself is fragmented throughout physical memory. For example, a title could allocate 14 MB of memory and address it as a contiguous address space, even though there may not be 14 MB of contiguous RAM available. Though physical RAM itself may be fragmented, titles can still allocate as much total RAM as is available and not worry about the fragmentation.

Heap Memory

The heap functions enable a title to create and access a private heap, a block of one or more pages of virtual memory in the address space of the title. The title can then use the heap functions (such as HeapAlloc) to manage the memory in that heap. A heap is designed and optimized for the allocation and deallocation of small blocks of memory (typically, smaller than the system page size of 4KB), but allocations of larger blocks can also be performed. From a system perspective, there is no difference between memory allocated from a private heap and that allocated by using the virtual memory allocation functions. The heap functions provide a structured layer above the virtual memory functions. Heaps allocate virtual memory using the virtual memory functions, and then manage smaller allocations and deallocations within that section of virtual memory. The memory management routines in the C runtime libraries are built on top of the heap functions themselves, so it is more efficient to call the heap functions directly rather than relying on CRT allocation routines.

Physical Memory

As mentioned above when discussing virtual memory, a contiguous range of virtual memory may not be backed by a contiguous range of physical memory. Adjacent, committed pages in the virtual address space may be backed by two pages of RAM located in completely different parts of the 64 MB physical memory. Often, when dealing with video or other hardware, it is beneficial (or even necessary) for a continguous range of physical memory to be used. For example, a DMA transfer may require that all memory blocks used in a single DMA operation be adjacent. The kernel provides support for allocating contiguous physical memory when required. Again, as with virtual memory, this kernel support is made available through XTL routines, in this case the XPhysicalxxx functions, such as XPhysicalAlloc. When requesting physical memory, the allocation request will fail if the requested amount of RAM is not available in one contiguous section of free physical memory.

Demand Paging

The kernel does not perform demand paging. In demand paging, the contents of a portion of physical memory are written to the hard disk and then that physical memory is freed for use by another process (or another portion of the same process). Later, when the first process attempts to access that memory again, the original data is read back from the hard drive and once again stored in physical memory.

Writing to the hard disk is a time-consuming task compared to reading data directly from memory. Demand paging sacrifices performance so that multiple processes can allocate more memory than is available, and not worry (as much) about the effects on other processes or the system software itself.

The Xbox kernel makes the opposite choice, favoring performance. Because only a single process is running at a time, demand paging would offer little benefit and would inflict performance losses and periodic interrupts on the title. Instead, a title must keep track of the amount of memory it allocates and avoid requesting more memory than is physically available. Memory allocations will begin to fail after all available physical RAM (64 MB) has been allocated.

Xbox Development System Memory

Xbox development and debug consoles have an extra "upper" 64 MB of physical RAM that can be used while developing and debugging titles. This upper memory is not available on retail Xbox consoles. (So game titles should not rely on it.) For testing purposes, this upper bank can be disabled with the ImageBld switch /LIMITMEM.

The upper memory (if not disabled) will be used by the virtual and heap memory allocation functions to satisfy memory requests after all memory in the lower 64 MB has been allocated. For example, a call to VirtualAlloc or HeapAlloc may, once the lower bank is full, return a virtual address backed by physical RAM residing in the upper memory bank. Additionally, virtual or heap memory backed by RAM in the lower bank may be moved into the upper bank to satisfy requests specifically for memory in the lower bank.

The physical memory allocation functions (XPhysicalAlloc and D3D_AllocContiguousMemory) will allocate memory only from the lower bank, regardless of whether /LIMITMEM is specified, and will fail if the requested amount of memory cannot be obtained from the 64 MB lower memory bank. As mentioned above, virtual or heap memory backed by RAM in the lower bank will be moved to the upper bank (if it is not disabled) in order to satisfy requests for physical memory in the lower bank.

The debug manager and debug manager routines, used for debugging and remote access on development and debug consoles, will allocate memory only from the upper bank, regardless of whether /LIMITMEM is specified. The DmAllocatePool function will always allocate and return memory from the upper 64 MB of RAM. Debug and development tools (and the title's own debug routines) can thus allocate memory without interfering with the title's normal allocations in the lower bank.
