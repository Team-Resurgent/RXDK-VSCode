Notifications and State Information

The notifications service is used to notify titles of real-time events. The list of possible events is currently restricted to friend requests and game invitations, but the service might be extended in the future to support a wider variety of notifications. The notification system is designed primarily for service-to-Xbox communication. It is not expected to support any form of real-time messaging.

Notifications are an automatic part of the underlying network traffic between the Xbox secure network stack and the Xbox Live service. On the client Xbox, incoming notifications are queued by the low-level network stack and are retrieved by the title code with the XOnlineNotification family of functions.

None of the notification functions require an XOnlineFriendsStartup task or XOnlineFriendsEnumerate task to be active.

Notifications

The notifications can be interpreted by the title by means of the XOnlineGetNotification or XOnlineGetNotificationEx functions, which allow Boolean conditions to be checked. This check is fast, does not generate any network traffic, and depends only on the XOnlineLogon task handle being open. Hence, it is practical to use during gameplay.

New titles should use XOnlineGetNotificationEx rather than XOnlineGetNotification. XOnlineGetNotificationEx provides access to notifications for new features such as teams. When a message comes in, XOnlineGetNotificationEx passes the notification type, ID, and some flags through the XONLINE_NOTIFICATION_EX_INFO structure pointed to by the pNotificationInfo parameter. If you want your title to take action on the notification or get more information, have it call one of the XOnlineMessage family of functions.

If there are more notifications, XOnlineGetNotificationEx sends back XONLINE_NOTIFICATION_STATE_FLAG_MORE_ITEMS through the pdwStateFlags parameter. At that point, your title should call XOnlineGetNotificationEx again to retrieve the remaining notifications. Because the notification queue is set to a fixed size, it is possible (but unlikely) for it to overflow. If too many notifications occur between calls, XOnlineGetNotificationEx sets pdwStateFlags to XONLINE_NOTIFICATION_STATE_FLAG_OVERFLOW_ITEMS. In this event, your title should call XOnlineMessageEnumerate to retrieve the current state.

XOnlineGetNotificationEx also enables titles to determine the state of message retrieval. The logon task completes prior to the full messaging list being available. The download continues in subsequent logon task continue calls. Until the specified subscriber's list is ready to use, the XOnlineGetNotificationEx passes back the value XONLINE_NOTIFICATION_STATE_FLAG_PENDING_SYNC through its pdwStateFlags parameter. The message list is also empty. When the logon task completes, titles can be sure that an empty list really means the user doesn't have any messages.

Notification Type Description
XONLINE_NOTIFICATION_FRIEND_REQUEST Returns true if any new friend requests have been received from other players since either: 1) the last call to XOnlineLogon, or 2) the XOnlineFriendsEnumerate task was last closed. Titles can use this to detect the first new friends request received during gameplay, or to indicate a new friends request is present when displaying a menu.
XONLINE_NOTIFICATION_GAME_INVITE Returns true if any new game invitations have been received from other players since either: 1) the last call to XOnlineLogon, or 2) the XOnlineFriendsEnumerate task was last closed. Titles can use this to detect the first new game invitation received during gameplay, or to indicate a game invitation is present when displaying a menu.
XONLINE_NOTIFICATION_NEW_GAME_INVITE Returns true if any new game invitation has been received from a friend since the last time XOnlineGetNotificationEx was called. Titles can use this to detect each invitation received throughout gameplay, so the player can be notified by means of a glyph.

Currently, all notifications are only actionable by means of enumerating the friends list.

Presence Information

Player state can be set by means of the XOnlineNotificationSetState function. This function does not immediately create any network activity, so it is practical to call during gameplay. The information specified by this function is communicated through the online heartbeat, which is processed by the task created by XOnlineLogon. Redundant and rapid changes are consolidated throughout the system. Because of this, state information can take several seconds to propagate to a player's friends, and should not be used if more current information is available by means of the current game session. State information is split into three pieces: state flags, session ID, and state data.

State flags communicate core information about a player's state across all games by means of the friends list.

State Flag Description
XONLINE_FRIENDSTATE_FLAG_ONLINE Indicates if the player should appear online to his or her friends. If this flag is cleared, the player will appear offline to each of his or her friends. The player will appear offline to each of his or her friends unless the player sends an invitation to his or her friends, or accepts an invitation from them. Titles can have a player appear offline at either the player's request or if the player is busy with non-game activity (such as downloading new content).
XONLINE_FRIENDSTATE_FLAG_PLAYING Indicates if the player is in an active game session (as opposed to being in a lobby or navigating menus, and so on). Specific interpretation of this state flag is left up to individual titles.
XONLINE_FRIENDSTATE_FLAG_VOICE Indicates whether the player's voice peripheral is active. This state flag must be clear if the peripheral is inactive or if the player's voice has been suspended by policy.
XONLINE_FRIENDSTATE_FLAG_JOINABLE Indicates the player's session can be joined by his or her friends without needing an invitation. To avoid lots of friends racing for the same slot, it is recommended that titles set this state flag only if there are a large number of open slots for a game.

The sessionID parameter identifies the matchmaking session the player is currently in. If the player is not in a session, it must be zeroed out. The session ID must be updated when the player enters, creates, re-creates, or exits a game. It is used by the player's friends to determine whether game invitations are still current. Furthermore, it is used in combination with the XONLINE_FRIENDSTATE_FLAG_JOINABLE state flag to communicate which matchmaking session to join.

Finally, the pbStateData buffer is 8 bytes that titles can use to encode title-specific data. Examples include current score, time remaining in game, character status (for example, surrounded by aliens and about to die), and character location.
