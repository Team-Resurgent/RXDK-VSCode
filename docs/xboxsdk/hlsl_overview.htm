High-Level Shader Language (HLSL)

The High-Level Shader Language (HLSL) for Xbox allows developers to use a C-like language to generate vertex shaders.

HLSL supports the development of shaders from C-like functions. The language supports many standard language features such as functions, expressions, statements, standard data types, user-designed data types, and preprocessor directives.

The Xbox HLSL Compiler (XFXC) is used to compile shader programs into the microcode format used by the Xbox game system.

Note    The targets supported by the compiler are "xvs_1_1" for a vertex shader and "xvsv_1_1" for a vertex shader that outputs a screen space vertex position (#pragma screenspace). The pixel shader target "xps_1_1" is currently unsupported.

Semantics

Semantics are used to bind variables declared in HLSL code to vertex shader stream inputs and to constant registers declared in game code. They are also used to bind vertex shader output variables declared in HLSL code to pixel shader inputs. A constant register can be associated with a variable using the ": register(cn)" semantic, where n is the index of the constant register to associate. For example, a global variable containing the direction of an infinite light source that will be loaded into constant register 8 is declared as follows:

float3 LightDirection : register(c8);

If the variable type is such that the value will span multiple registers, it is assumed that the associated registers begin with the index defined in the semantic and continue in a contiguous manner. For example, the following matrix is associated with constant registers c0 through c3:

float4x4 WorldViewProj : register(c0);

Vertex stream input semantics may similarly be declared using the ": register(vn)" semantic, where n ranges from 0 through 15. A vertex normal element mapped to v1 by a vertex shader declaration could be associated with the "Normal" input variable in the source code as follows:

VS_OUTPUT ShadeBlobVertex
(
   const float3 Normal   : register(v1)
   ...
)
{
   ...
}

Alternately, vertex stream inputs may also be associated using more familiar named semantics. These semantics are as follows:

  • POSITION
  • BLENDWEIGHT
  • NORMAL
  • DIFFUSE
  • SPECULAR
  • BACKDIFFUSE
  • BACKSPECULAR
  • FOG
  • TEXCOORD0
  • TEXCOORD1
  • TEXCOORD2
  • TEXCOORD3

Use of one of these semantics with a variable will cause the variable to be associated with the stream input defined by the corresponding D3DVSDE_x defined in d3d8types.h (that is, BACKDIFFUSE corresponds to D3DVSDE_BACKDIFFUSE, which causes an association with v register 7).

Vertex shader outputs are associated with pixel shader inputs using familiarly named semantics as well. These semantics, along with their corresponding vertex shader output registers and pixel shader input registers, are as follows:

  • POSITION - oPos
  • PSIZE - oPts
  • FOG - oFog
  • COLOR0 - oD0 / v0
  • COLOR1 - oD1 / v1
  • BACKCOLOR0 - oB0 / v0
  • BACKCOLOR1 - oB1 / v1
  • TEXCOORD0 - oT0 / t0
  • TEXCOORD1 - oT1 / t1
  • TEXCOORD2 - oT2 / t2
  • TEXCOORD3 - oT3 / t3

Explicit Constant Values

It is worth noting that any constants whose values are explicitly given in the source code will still be associated with constant registers through 'def' instructions in the generated vertex shader assembly. For example, the following source

ScaledRadius = Radius * 20.0f;
Output.Position = float4(Input.Position * ScaledRadius, 1.0f);

would lead to assembly code containing one or more instructions similar to the following:

def c[0], 1, 0, 0, 20

Since the assembled vertex shader microcode does not store the values of constant registers declared using 'def', it is important that these values and their associated constant registers be preserved and correctly set in the game code. Since the ID3DXConstantTable interface and D3DXSHADER_CONSTANTTABLE are present only in DirectX 9.0, the best way to do this on Xbox is to pass a pointer to an XGBuffer into the pConstants parameter of XGCompileShader (or to use the /Fc option in the XFXC tool) and to save the returned buffer. Later, when the vertex shader is created in the game, take the contents of the pConstants buffer, append the desired vertex shader declaration, and pass the result as the new pDeclaration into CreateVertexShader. In this way, all explicit constant values used in the HLSL source will then be applied before running the shader without ever having to track exactly what values were used or into which constant registers they were placed.

Intrinsics

Intrinsic functions are currently unsupported.
