Occlusion Culling

Z-Occlusion Culling

The Xbox GPU supports optional z-occlusion culling. This culling takes place right before the per-pixel pipelines run and is done in addition to any normal culling that occurs using D3DRS_ZENABLE. The z-occlusion culling occurs by testing multiple pixels at once, either rejecting them all, or letting them all pass. It is possible (very likely) that some of the pixels passed on are actually occluded. Although z-occlusion testing is on by default, it may be disabled (or enabled) using the D3DRS_OCCLUSIONCULLENABLE render state.

Normal per-pixel z-rejection must be enabled using D3DRS_ZENABLE before z-occlusion culling can take effect.

When both per-pixel z-rejection and z-occlusion culling are enabled, the depth buffer is read twice, which can result in a bandwidth penalty for non-occluded pixels. This is generally more than balanced by the texture reads and pipeline processing saved by the z-occlusion culling, but it is nonetheless a potentially substantial cost. The more highly compressed a depth buffer is (using tiled memory), the more effective z-occlusion culling is at reducing memory bandwidth demands.

Z-occlusion culling can be enabled for all packets (a 64-byte chunk of depth buffer) or for compressed packets only. The greater the depth complexity of a scene, the more likely it is that enabling fast-path occlusion culling for all packets will result in increased performance. By default, z-occlusion culling is enabled for all packets, but this can be modified using the D3DRS_DONTCULLUNCOMPRESSED render state. Note that when multisampling is enabled, the hardware automatically limits occlusion testing to compressed packets only, regardless of the render state set using D3DRS_DONTCULLUNCOMPRESSED.

Note   With a 16-bit (D16) depth buffer, errors have been observed with z-occlusion culling turned on. Occasionally, pixels drawn at the same z-value as was previously in the depth buffer will be rejected even though the z comparison function is set to accept equal values. For example, when using two-pass rendering this would result in some pixels from the first pass showing through the second pass. If this occurs, known solutions are to turn D3DRS_OCCLUSIONCULLENABLE off or to use a 24-bit depth buffer.

Fast-Path Stencil-Occlusion Culling

The Xbox GPU also supports stencil-occlusion culling. Just as with the z-occlusion culling, stencil-occlusion culling occurs before the per-pipeline is run and is done by testing multiple pixels at once, either rejecting them all or letting them all pass. If a pixel is allowed to pass, there will be a re-read of the stencil values later in the pipeline. Stencil-occlusion culling is on by default, but it may be disabled (or enabled) using the D3DRS_STENCILCULLENABLE render state.

For stencil-occlusion culling to take effect, normal stencil rejection must be enabled using D3DRS_STENCILENABLE, z-occlusion must be enabled using D3DRS_OCCLUSIONCULLENABLE, z-buffering must be on, and D3DRS_ZFUNC cannot be D3DCMP_ALWAYS.

The render state D3DRS_DONTCULLUNCOMPRESSED applies to both stencil-occlusion and z-occlusion culling.

Note   If both z-occlusion and stencil-occlusion culling are enabled at the same time, they will operate off of the same reads. It's not as if one read is needed for z culling and another for stencil culling; there is at most only one occlusion read per pixel, which is shared if both z and stencil occlusion are active.
