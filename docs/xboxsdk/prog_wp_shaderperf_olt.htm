Pixel Shader Performance One Last Time (I Hope!)

In the book The Mythical Man-Month, Frederick Brooks describes how products evolve through successive versions. For the first version, the team knows they are venturing into uncharted waters, so they keep the project's scope under control, proceeding carefully and producing a product that is modest and flawed, but not disastrously so. For the second version, the team, now feeling that they're experts, aim for big things—and as a consequence, do produce a disaster. Only with the third version does a combination of experience and humility allow a truly excellent product to be produced.

This illustrates a generally useful principle: everything interesting involves a learning curve, and sometimes it takes a long time to climb that curve.

I tell you this by way of introduction to the topic of pixel shader performance. You see, it's been a long learning curve for me, one which unfortunately you've seen in action in previous Xbox white papers. I think, however, that we've finally reached the end of the trail.

In this paper, I will tell you two things: 1) everything I know about pixel shader performance, and 2) the fact that I don't know enough—pixel shader performance is so complex that it's not statically analyzable, but fortunately you can run the pixel shader performance analysis program FILLRATE and just measure how fast any pixel shader configuration is.

Before we begin, I want to point out that the pixel shader consists of two parts, texture stages and register combiners. Register combiner performance is simple: if you use 1 or 2 of them, it takes 1 cycle; 3 or 4 is 2 cycles; 5 or 6 is 3 cycles; and 7 or 8 is 4 cycles. The register combiners and texture stages coprocess, so the time the pixel shader takes to run is the larger of the texture stage and register combiner times for any particular configuration, not their sum.

Beyond that, there's nothing to say about register combiner performance; the remainder of this paper will discuss texture-stage performance.

The basic nature of texture-stage performance

The basic nature of texture-stage performance is, in a word, complicated. There are a lot of complex internal states, and according to NVIDIA engineers, the only way to describe it would be a complete description of the texture-stage hardware, which for all practical purposes means that there's no way to describe the performance rules for the texture stages. (I must admit that, as this paper illustrates, it is possible to come up with rules that explain quite a bit about texture-stage performance, but as we'll see, there's quite a bit that's still a mystery.)

Fortunately, it's not necessary to describe those rules, because you can just run the FILLRATE program with any texture-stage configuration you want and measure its performance on the actual hardware. However, there are a lot of possible ways to set up the texture stages, so in order to help you out a bit with searching that space for high-performance configurations, I'll tell you what I do know about texture-stage performance. However, this will not be a hard-and-fast, down-to-the-metal description, both because that's not practical and because I don't have that level of knowledge. I'm just trying to give you an idea of how texture-stage performance works in general, and what sorts of factors might cause performance to jump or lag, but bear in mind that for the most part I'm going to be presenting you with guiding examples rather than definitive rules.

One-pass isn't the same as one-cycle, because of the following rule: a maximum of two texture fetches may be performed in a single cycle. Therefore, any configuration of three or four PROJECT2D textures will take 2 cycles. (Actually, it will take a minimum of 2 cycles; texture fetch time may cause it to take longer. Basically, if the required texels are in the texture cache, fetch time is zero, but if they're not in the texture cache, the time to fetch them from memory is variable and can be many cycles. For the purposes of this paper, however, I'll just talk about texture-stage cycles, disregarding fetch time, with the understanding that these are minimum times.)

Multiple passes

 

If one or more texture stages are dependent on the output of other texture stages, then they obviously cannot run at the same time. The pixel shader handles this by putting dependent operations in separate passes that run after the passes theyre dependent on. So, for example, the following configuration runs in two passes:

 

Texture stage 0

Texture stage 1

Texture stage 2

Texture stage 3

PROJECT2D

BUMPENVMAP

---

---

 

The PROJECT2D texture runs on the first pass, and then the BUMPENVMAP texture runs on the second pass, using the output of the first pass as input. Each pass takes a minimum of 1 cycle, so this configuration runs in 2 cycles.

 

This is a convenient time to point out that grouping into passes doesnt depend on the ordering of the texture stages. Consider the following configuration:

 

Texture stage 0

Texture stage 1

Texture stage 2

Texture stage 3

PROJECT2D

BUMPENVMAP

---

PROJECT2D

 

The first pass includes all non-dependent textures, in this case meaning the two PROJECT2D textures, which run together in 1 cycle as usual. The second pass is the BUMPENVMAP texture, which runs in 1 cycle. Thus, the total time for this configuration is just 2 cycles, rather than the 3 cycles you would expect if textures had to run in order of texture-stage number.

 

This is also a good time to mention that the above cases don't run in exactly 2 cycles; it's actually more like 2.2 cycles. The reason for this is that some internal circuitry does double duty to convey texcoords to dependent textures, resulting in a slowdown of roughly 10% whenever dependent textures are used. Thus, you won't see integral cycle times when dependent textures are in use.

Stepping back

 

So is it just that simple? Figure out the number of passes and the pairing and you're all set? Heck no! What I've described is a generally useful way of thinking about texture stage performance, but it's just an approximation, and doesn't adequately describe a number of cases. For example, consider this earlier case:

 

Texture stage 0

Texture stage 1

Texture stage 2

Texture stage 3

PROJECT2D

BUMPENVMAP

BUMPENVMAP(1)

BUMPENVMAP(2)

 

This takes 8 cycles, which would imply that the 4 passes (because of serial dependence) each take 2 cycles. But why would they take 2 cycles, when we know from the simple case of PROJECT2D followed by BUMPENVMAP that each texture can run in 1 cycle?

 

The answer is that at a certain level of overall texture-stage complexity, the internal pipeline can't store and forward all the data for each pass in a single cycle, and requires 2 cycles per pass. This is what's happening in the above configuration, so each of the 4 passes takes 2 cycles, for a total of 8. The problem is, I don't have a crisp definition of what a certain level of overall texture-stage complexity means. All I can tell you is that you should be aware that if you use enough texture stages with sufficiently complex texture ops, you're likely to see your passes take at least 2 cycles.

 

As a 3-cycle-per-pass example, consider the following configuration:

 

Texture stage 0

Texture stage 1

Texture stage 2

Texture stage 3

PROJECT2D

BUMPENVMAP

BUMPENVMAP(0)

BUMPENVMAP(0)

 

This has two passes, but takes 6 cycles; the three BUMPENVMAPs run together in one pass, taking 3 cycles, so the PROJECT2D takes 3 cycles, too (since all passes have to take the same amount of time).

 

As another example, consider this configuration:

 

Texture stage 0

Texture stage 1

Texture stage 2

Texture stage 3

PROJECT2D

BUMPENVMAP_LUM

BUMPENVMAP_LUM(1)

---

 

This runs in 8 cycles. It seems as if this configuration would take five passes (one for PROJECT2D, two for the first BUMPENVMAP_LUM, two for the second BUMPENVMAP_LUM, because its dependent on the first). However, there's no way that five equal-length passes can take 8 cycles.

As it turns out, however, the way the hardware works is that only the BUMPENVMAP result from stage 1 is passed on to stage 2. Therefore, there are actually only three dependencies, resulting in only four passes, not five, and four passes neatly produces 8 cycles with 2 cycles per pass; probably the two luminance calculations are performed together on the last pass, or perhaps one luminance is performed during the third pass, along with the second BUMPENVMAP. So this case is readily explainable. But now consider this:

 

Texture stage 0

Texture stage 1

Texture stage 2

Texture stage 3

PROJECT2D

BUMPENVMAP_LUM

BUMPENVMAP_LUM(1)

BUMPENVMAP_LUM(2)

 

which runs in 12 cycles. Here there would seem to be five passes: the initial PROJECT2D, the chain of three dependent BUMPENVMAPs, and finally either the three luminance calculations in a single pass together, or two of them in the same pass with two of the BUMPENVMAPs, and the third in a separate pass of its own. However, in order to produce 12 cycles, there would have to be either four 3-cycle passes or six 2-cycle passes. It looks like two of the luminance calculations are done in one pass, and the other one in a separate pass, but that's just a guess. The important thing is that this nicely illustrates the point that the internal pipeline is often inscrutable, and that FILLRATE is the only sure way to evaluate the performance of any complicated texture-stage configuration.

 

A couple of notes

 

According to NVIDIA, many texture-stage configurations have special-case performance optimizations. These depend not only on texture ops (which ones, in which stages, and in what combinations), but also in some cases on renderstates. I don't have a list of such renderstates to give you, but as an example, NVIDIA engineers have mentioned that both fog and projective texturing can affect texture-stage performance.

 

Be aware that although PASSTHRU or CLIPPLANE by itself runs in 1 cycle, each additional PASSTHRU or CLIPPLANE takes 1 additional cycle, so a four-PASSTHRU configuration (or any combination of four PASSTHRUs and CLIPPLANEs) takes 4 cycles to run.

 

There's an additional note of interest concerning CLIPPLANE. When CLIPPLANE succeeds in clipping a pixel, the later stages of the pixel shader still run at their normal speed; that is, if you have CLIPPLANE followed by 8 register combiner stages, the pixel shader still takes 4 cycles even when the pixel is clipped. However, when CLIPPLANE clips a pixel, it does suppress texel fetches by later texture stages, which can result in improved performance due to more efficient use of the texture cache and less fetching of texels from memory.

 

When you run FILLRATE, make sure your FVF specifies texcoords for each stage that's set to a texture op that requires texels. Otherwise, your timings will be slower than they should be. Also, any texture stage set to DOT_ZW should have texcoords as input, although I don't know why it's needed. The simplest way to do this is to always set up texcoords for all four stages. This avoids all possible problems and has no performance cost. (This is the default setting; so you shouldn't have to do anything unless you change that part of the .ini file.)
