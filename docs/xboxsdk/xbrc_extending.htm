Extending XbRC

Extending the XbRC to understand new data types is quite simple:

  1. Adding a new class. Add a new class to represent your data type derived from XBRCRawData.
  2. Writing Tag Handlers. Write the tag handlers for the new data type as members of CXBParser. These are the Begin_tag and End_tag functions described in Parsing Using CXBParser.
  3. Adding Tag Handlers. Add the tag handlers to the startElement and endElement member functions of the CXBParser class, being sure to include them in the correct context.
  4. Writing the Constructor. Write the constructor that takes the parsed data and adds it to the class derived from XBRCRawData above.
  5. Writing the addData Function. Write the addData function for the class derived from XBRCRawData and call it from the CXBParser::characters member function based on the correct context.
  6. Writing needsCompiling function. Write the needsCompiling function for the class derived from XBRCRawData to determine if the newly parsed data matches the data in the intermediate file.
  7. Writing a Compile Function Write the compiler member function for the class derived from XBRCRawData.
  8. Storing Data for Dependency Checking. Write the inputIntermediateFile and outputIntermediateFile functions for the class derived from XBRCRawData to store the data for dependency checking.
  9. Writing an Optimize Function. Write the optimize function for the class derived from XBRCRawData to optimize the data, if desired.
  10. Writing a Link Function. Write the link function that outputs the data from the memory footprint or intermediate file to the final file format needed for the resource(s).

The CXBParser class drives the process and provides error handling if the parsing, compiling, optimizing, or linking run into problems.

When XbRC starts up, it parses the command line, starts up the SAX parser, and creates an instance of the CXBParser class. Based on the command line, it hands the file specified to the CXBParser class and starts parsing. When parsing is complete, XbRC makes sure no error was thrown and then starts compiling. After compiling and making sure the compile didn't throw any errors, it starts optimizing. After optimizing and making sure the optimization pass didn't throw any errors, it links the data into a resource file or files. It then exits with the error code of S_OK if everything went OK, or the error code of the problem encountered if not.

The CXBParser class is what runs the show. It provides the callbacks used by the SAX parser to hand us the data it parses from the XML file. It also knows about each resource type and does the specific work needed to create the containers for each resource type. CXBParser creates the containers and pushes them into two lists. The first is the master list that contains all of the data parsed out of the class. The second acts like a stack so that nested tags will be parsed correctly. The second list shows the current context that you are parsing. For example, if your current context is a cube map resource (that is, the top of the stack or second list is a resource of type cube map) and you encounter a texture tag, the context allows you to add the texture to the cube map and not to the general list of resources.

When you compile, optimize, or link, you are simply iterating through the master list of resources and calling their compile, optimize, and link functions to actually output the data. That's all there is to it.

The data containers are derived from the XBRCRawData class. Here is the header for this class:

//-----------------------------------------------------------------------------
// Name: XBRCRawData
// Desc: Base class from which all containers are derived.
// DEVELOPERS: All custom data types should derive from this.
//-----------------------------------------------------------------------------
class XBRCRawData
{
    protected:
        XBRCResourceType    m_dwResourceType;

    public:
        XBRCRawData()
        {
            m_dwResourceType = XBRC_INVALID_RESOURCE;
        }

        XBRCRawData( ISAXAttributes *pAttributes );

        virtual ~XBRCRawData()
        {
            m_dwResourceType = XBRC_INVALID_RESOURCE;
        }

        //Adds data to the class from XML file.
        virtual HRESULT addData( WCHAR *pwchChars, int cchChars ) = 0;

        // Compiles resource into binary format.
        virtual HRESULT compile() = 0;

        // optimizes binary format.  (Stripify triangles, etc.)
        virtual HRESULT optimize() = 0;

        // Writes intermediate file containing compiled/optimized binary data.
        virtual HRESULT outputIntermediateFile() = 0;

        // Reads intermediate file containing compiled/optimized binary data.
        virtual HRESULT inputIntermediateFile() = 0;

        // Checks dependencies to see if resource needs to be compiled.
        virtual BOOL needsCompiling() = 0;

        // links resource into final bundled format.
        virtual HRESULT link() = 0;

        virtual XBRCResourceType getResourceType()
        {
            // Return INVALID for the base class.  Nothing should ever
            // be this type of resource.
            return m_dwResourceType;
        }
};

All of the containers used in XbRC should be derived from this class.
