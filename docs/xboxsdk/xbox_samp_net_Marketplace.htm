Marketplace Sample

Demonstrates proximity-based conversations for large numbers of users over speaker and headset. It also shows a method for dynamically allocating bandwidth using simple peer prediction for a scalable mix of peer-to-peer and player-server voice traffic.

Overview

This sample illustrates the concepts in the voice partitioning white paper. The technical areas it covers are:

  • Speaker and headset output through XHV.
  • Coalescing voice packets to save bandwidth.
  • Proximity-based voice partitioning with low server overhead.

This sample does not comply with logon and other Xbox Live TCRs, because it demonstrates a way to do voice partitioning, not a full Xbox Live game.

Basic Problem

Partitioning voice is hard; not only do you have to limit how many other players you are sending to, but how many other players are sending to them. For instance, lets say you partitioned based solely on the n closest other players. If you have a player standing in the middle of a circle of people, and several other players in the circle are talking, the person in the center will have his bandwidth saturated with voice packets; because he is close to them all, he'll be on all of their receiving lists. 

         P   P!
      P!        P
          
     P     o     P!
     
      P!        P
         P   P!

You could have the server calculate all of the player-to-player routes, but this is costly in bandwidth and CPU cycles and, when players start talking, will result in a significant delay while the routes are recalculated. If they are precalculated, the middle player will be unable to hear someone who isn't in the closest four people, even if none of the closest four are talking.

Presented Solution

The solution demonstrated in the Marketplace sample involves a single bit per player being sent on the update packet from the server. This bit specifies whether a given player is talking or not. The player-to-player voice routes are completely calculated on each player based on the talking bits, so oversaturating a player can be avoided. Even if the update comes a little late, oversaturating a player will not last more than a single network update (50 ms by default), which gives adequate time to recover.

A priority is determined for each proximal player-to-player pair, and then bandwidth is allocated round-robin, with a player being removed from the send or receive set as soon as her bandwidth quota is used up. This quota could be scaled easily, based on players' QoS bandwidth numbers, so that some players would automatically receive or send more data than others based on what their connection would allow.

Priority is given to players near other players, players who aren't moving (and thus are likely to be in an established conversation), and players facing other players. Additional priority modifiers could be implemented, including higher priority for friends, etc.

Remarks

Because of the flexibility in network architecture required (support for both peer-to-peer and full server-forward for demonstration purposes), not much code here that can be reused in your games. Also, because of the added complexity in making a visual demonstration (bots, sorted sprites, soundbit editor for the bots), a lot of code here is not directly applicable to voice partitioning. The full voice path code is spread throughout the sample. Most of the partitioning code is in VoiceMgr.cpp, but some other important concepts, such as 3D sound buffer caching are implemented in Audio.cpp.

Doing a full build in Visual Studio will copy the media files to your Xbox.

Using the Sample

This sample requires a controller with a communicator plugged into port 0.

After you log on, you will be presented with four options.

  1. Host a game with bots, which simulate other players.
  2. Host a game without bots, enabling you to play with your friends.
  3. Join an existing game. The sample finds the best game based on QoS- no selection.
  4. Edit bot voices by recording some voice clips for the bots to use.

The sample's user interface supports the following input.

Move around.
X
Join private channel (through headset).
A
Talk on private channel (hold to talk).
Y

Host menu (if you are the host). The host menu lets you set:

  • Server-forward or peer-to-peer voice mode.
  • Network send rate.
  • Voice send rate.
  • Speech timeout.
  • Max. number of people you can listen to (downstream # of peers).
  • Max. number of people you can send to (upstream # of peers).

In the upper right corner of the Marketplace is a podium. If you stand behind it, voice will go through the server out to everyone in the Marketplace.

Sound Around the Marketplace

  • There is a localized 3D source at the fountain.
  • There is an ambient crowd sound that is scaled based on the total number of people talking.
  • If someone is talking but you can't hear them (because of network allocation), you will hear a walla-walla-walla sound emanating from their position.
  • If someone is talking and is sending traffic to you, you will hear his voice.
  • When a new person enters the Marketplace, you will hear a chime.

Programming Notes

The network stats (accessed by pressing the White button) shows only the traffic used for voice. If voice is piggybacked on a data packet, the 44 bytes for the packet header are not counted. If the voice is sent by itself (as in peer-to-peer), the 44 bytes are counted.

In peer-to-peer mode, the host bandwidth shows only the stats for voice going through the server; there would be no players locally in a massively-multiplayer game, and the stats are intended to show the impact on server bandwidth for using this method of voice.
