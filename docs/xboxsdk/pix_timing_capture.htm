PIX Timing Capture

The code that captures the timing data is part of the debug and profile version of the Direct3D library, but it is compiled out of the release version. PIX can provide the most accurate and relevant timing information when it is used to capture timing data from a profile version of a game. A profile version eliminates nearly all of the debug and parameter checking code from Direct3D, and causes it to run very nearly the same way it runs in a release version game.

All of the CPU timing capture code uses the CPU timer returned by QueryPerformanceCounter. The time is sampled at the start and at the end of the function.

Some CPU timing events have associated GPU events. For these events, commands are written to the push-buffer at the start of the function call to measure the GPU start time. A command is also written to the push-buffer at the end of the function call to measure the GPU end time. The capture code inserts a callback command using InsertCallback to mark the beginning of the GPU event. This command will cause an interrupt to be generated when the GPU encounters it in the push-buffer. The time of that interrupt is then stored as the beginning of the GPU event. At the end of the function call, a WritePixelCount command is inserted into the push-buffer. This causes the GPU to write the current pixel-count register value into main memory along with a timestamp. This method of measuring the end of a GPU command is better than using a write callback as the previous D3DPerf APIs did, because it does not cause a back-end flush. The GPU writes the 4-DWORD pixel count value directly into the capture buffer. It is up to the timing capture code to determine the correspondence between the CPU clock that is used for the begin GPU event measurement and the GPU clock that is used by the WritePixelCount command to measure the end of the GPU event.
