Palettes and Push, Vertex, and Index Buffers

The resource headers for IDirect3DPalette8, IDirect3DPushBuffer8, IDirect3DIndexBuffer8, and IDirect3DVertexBuffer8 are all very similar and are defined as below:

struct _ResourceName {
  // Resource-specific declarations
  // ...
  
  DWORD Common;
  DWORD Data;
  DWORD Lock;
};

Members

Common
Holds all of the information common to all resources inheriting from IDirect3DResource8. Information is stored in the following bit fields.
Field Bits Description
REFCOUNT 16 Reference count for the resource, which will be destroyed if this value ever goes to zero.
TYPE 3 Holds the type of the resource. Depending on the resource, this will be one of the following:
  • D3DCOMMON_TYPE_INDEXBUFFER
  • D3DCOMMON_TYPE_PALETTE
  • D3DCOMMON_TYPE_PUSHBUFFER
  • D3DCOMMON_TYPE_VERTEXBUFFER
INTREFCOUNT 4 Internal reference count that is used to determine whether IDirect3DDevice8 is using this resource.
VIDEOMEMORY 1 This bit is not used.
D3DCREATED 1 Flag to indicate that the resource was created using one of the Create methods of IDirect3DDevice8. This flag should not be set for manually created resources.

Palettes use the upper nibble of the Common field to store the size of the palette. If Size is the D3DPALETTESIZE of the palette, this may be set using code similar to the following:

Common |= Size << D3DPALETTE_COMMON_PALETTESIZE_SHIFT;

Push-buffers have an additional flag that can be set on the Common field: D3DPUSHBUFFER_RUN_USING_CPU_COPY. This flag indicates that the data stored in the push-buffer resource should get copied into the graphics processor's push-buffer instead of going through a jump.

Data
The offset of the data for this resource in physical memory. The memory for the data of a resource must be stored in contiguous, write-combined memory, except for index buffers and some push-buffers. The data allocation for each resource must be aligned on a specific boundary depending on the resource type.

All palettes must be aligned using D3DPALETTE_ALIGNMENT. Palettes store their data as an array of 32-bit ARGB colors that is at least as long as the size indicated in the header's D3DPALETTESIZE field.

All push-buffers must be aligned using D3DPUSHBUFFER_ALIGNMENT. The data for a push-buffer consists of recorded push-buffer commands. The data for a push-buffer can be allocated two different ways. If D3DPUSHBUFFER_RUN_USING_CPU_COPY is set, the data needs to be allocated in noncontiguous cached memory so that it can be efficiently copied into the graphics processor's push-buffer. If that flag is not set, the push-buffer data must be stored in write-combined mapped contiguous memory so it can be executed in place.

All vertex buffers must be aligned using D3DVERTEXBUFFER_ALIGNMENT. The data allocation for a vertex buffer just contains an array of vertex information.

All index buffers must be aligned using D3DINDEXBUFFER_ALIGNMENT. The graphics processor has no direct support for a separate index buffer, so its contents must be copied into the push-buffer by the graphics driver, thus the memory used to store the indices should be noncontiguous cached memory. The data field in an index buffer contains a pointer to the virtual memory that contains the index buffer data instead of the normal offset in physical memory. The data for an index buffer contains an array of 16-bit indices.

The contiguous memory allocation ensures that the consecutive virtual pages of the allocation will use consecutive physical pages, because the graphics processor does not have any support for virtual memory. The memory must be write combined to ensure coherency between the graphics processor and the CPU. The write-combined mapping inhibits the CPU's normal caching behavior but will combine writes in short-term buffers to improve write speeds. Direct3D automatically flushes these buffers whenever any new commands are sent to the graphics processor.

Lock
Contains the fence that must be crossed before this resource is no longer used by the graphics processor. It will contain zero if there is no possibility of this resource being in use or if it is currently set on the device. This field should always start out as zero and should never be modified by the title.
