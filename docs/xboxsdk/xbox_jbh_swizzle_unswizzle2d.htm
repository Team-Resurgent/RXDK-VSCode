XGUnswizzleRect

Unswizzles a subrectangle from a texture into a buffer.

VOID WINAPI XGUnswizzleRect(
  LPCVOID pSource,
  DWORD Width,
  DWORD Height,
  LPCRECT pRect,
  LPVOID pDest,
  DWORD Pitch,
  CONST LPPOINT pPoint,
  DWORD BytesPerPixel
);

Parameters

pSource
The source texture. The source buffer needs to be 16-byte aligned.
Width
The width of the entire source texture.
Height
The height of the entire source texture.
pRect
The rectangle within the texture to copy.
pDest
The destination buffer. For speed reasons, this buffer should be at least 4-byte aligned.
Pitch
The pitch of the destination buffer.
pPoint
Where to copy the rectangle to.
BytesPerPixel
The number of bytes per pixel.

Return Values

None.

Remarks

If pRect is NULL, pPoint is NULL and Pitch == 0, this routine will assume that the source texture is exactly the same size as the destination buffer and will unswizzle the whole thing. This routine will run considerably faster in that case.

For optimal performance when doing partial-texture unswizzling, source rectangles should be aligned according to the table below:

Bytes-per-Texel of Source TextureAlignment of Source Rectangle
1 16x4
2 16x4
4 8x4

If the texture contains an aligned block, either 16x4 or 8x4, then pSource must be 16-byte aligned; however, for speed reasons, you should always 16-byte align both pSource and pDest, even though it is not required in all cases.

Unswizzling unaligned rectangles is several times slower than unswizzling aligned rectangles. It is almost always faster to unswizzle the aligned rectangle surrounding the area you need to unswizzle, rather than unswizzling the smaller unaligned rectangle. The exception to this is where you only need to work with less than a dozen texels.

This method has been optimized for write-combined memory.

Linear textures have a minimum pitch of 64 bytes, and their pitch must be a multiple of 64. If width*bpp is not a multiple of 64, padding is added to the end of each row, in order to have a multiple-of-64 pitch. However, memory buffers (created with new[] or malloc(), for example), do not have this pitch requirement. Since most swizzling and unswizzling will be performed using these memory buffers, XG-Swizzling routines are set up to work with them. If you pass 0 for the pitch, XGSwizzleRect will assume that the pitch of your linear buffer is width*bpp, with no padding. If you are in fact swizzling from a linear texture (D3DFMT_LIN_A8R8G8B8, for example), specify the pitch of that texture, otherwise strange graphical glitches may occur.
