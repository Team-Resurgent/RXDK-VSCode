XBldSigs Tool

The XBldSigs tool is used to build a content metadata file with computed signatures for the content data.

XBldSigs takes a collection of files and constructs a single metadata file that describes the content and contains content signatures. This process of generating the signatures and other metadata with XBldSigs is for offline titles. Xbox Live titles should use BuildOffer.

See Content Signatures for a discussion of the content signature scheme and how XBldSigs fits into that scheme.

Usage

USAGE: xbldsigs [options] dirroot
Option Description
-metadata:filename Input metadata filename. Contains the name of the content and any other metadata necessary for the Xbox dashboard to display the content in the saved games area. This file has the same format as a title's metadata file. This argument is required.
-titleid:titleid Title ID. Determines which title's save area the content will be installed to. Generally this information will be ignored, because most installers will explicitly indicate the title ID for which the content is being installed, but some installers (XbInstCont, for example) may observe this title ID if one is not otherwise specified. The default is zero.
-offerid:offerid Offering ID (32 bits). Specifies the identifier of the content offering. The default is zero.
-flags:contentflags Content filter flags. Specify title-specific flags describing the content. These flags can be used during content enumeration to filter the returned content. The default is zero.
-sigdll:file.dll Signature DLL. This option can be used multiple times to specify multiply signature DLLs. The default is that no signature DLLs are used.
-out:file.xbx Output filename. If you do not specify a specific name, the default is dirroot\contentmeta.xbx.

The dirroot argument specifies the directory that is the root of the content and is the default location for the output metdata file.

The tool will scan all of the files under the dirroot directory and compute signature information for each file. Signature information will be generated for each file by calling each one of the specified signature providers (the signature DLLs and the filelists), in order, until one of them indicates that it has calculated appropriate signature data for that file. If none of the signature providers recognizes the file, that file will receive a single, non-literal signature over its entire contents.

The XBldSigs tool creates metadata files for offline content only. That is, content that is delivered from the title DVD.

Signature DLLs

Signature DLLs provide title developers a way to determine how to partition file data into signatures without knowing beforehand what byte ranges are to be signed. Each signature DLL implements a single function:

BOOL
CALLBACK
CalculateSignatures(
	IN LPCSTR FileName,
	IN LPBYTE FileContents,
	IN DWORD FileSize
);

When a DLL returns TRUE, indicating that a file was processed, no further processing for that file occurs, and the tool moves to the next file. If the DLL returns FALSE, indicating that the file was not processed, the tool will call the next DLL or file list in sequence for processing.

A single function is provided to the DLL for registration of signature information:

BOOL
WINAPI
XRegisterSignature(
	IN DWORD IndexNumber,
	IN DWORD Flags,
	IN LPVOID SignatureData,
	IN DWORD SignatureSize,
	IN LPCSTR FileName,
	IN DWORD FileOffset,
	IN DWORD DataLength
);

IndexNumber is the slot number in the signature table that this signature will occupy. A value of -1 (0xFFFFFFFF) will cause XRegisterSignature to assign a slot number. All literal signatures must precede all non-literal signatures.

Flags is a combination of the following:
Value Description
XSIG_FLAG_AUTOMATIC Indicates an automatic signature.
XSIG_FLAG_NAMED Indicates a named signature.

SignatureData and SignatureSize specify the literal signature data to be inserted into the signature file. They are ignored for non-literal signatures.

FileName, FileOffset, and DataLength specify the data to be stored in the signature name table for named signatures. A data length of zero means all of the data from the specified offset to EOF. These arguments are ignored for anonymous literal signatures. Anonymous non-literal signatures must include this information so the signature can be correctly calculated at installation time.

The return value is TRUE if the function succeeds, or FALSE if it fails. GetLastError will be set if the function fails.

Signature DLLs are Win32 DLLs that should link with xcontent.lib and include xcontent.h.

Again, see Content Signatures for a discussion of the content signature scheme and the types of signatures used in that scheme.

A Sample Signature DLL

This pseudocode would be used for a content file that has a single TOC at the beginning of the file which specifies the chunks of data that make up the remainder of the file. It uses named, non-literal signatures:

BOOL CALLBACK CalculateSignatures(Name, Data, Size)
{
    FileExt = strstr(Name, ".abc");
    if(!FileExt || FileExt[5] != 0)
        return FALSE;

    GetTOCInfo(Data, &TOCData, &TOCSize);
    XRegisterSignature(-1, XSIG_FLAG_NAMED | XSIG_FLAG_AUTOMATIC, NULL, 0,
        Name, 0, TOCSize);
    for(index = 0; index < TOCData.NumberOfSections; ++index) {
        XRegisterSignature(-1, XSIG_FLAG_NAMED | XSIG_FLAG_AUTOMATIC, NULL, 0,
            Name, TOCData.Sections[index].Offset,
            TOCData.Sections[index].Size);
    }

    return TRUE;
}

This pseudocode might be used if the content consists of a single file, formatted as above. It uses anonymous, non-literal signatures:

BOOL CALLBACK CalculateSignatures(Name, Data, Size)
{
    FileExt = strstr(Name, ".abc");
    if(!FileExt || FileExt[5] != 0)
        return FALSE;

    GetTOCInfo(Data, &TOCData, &TOCSize);
    XRegisterSignature(0, XSIG_FLAG_AUTOMATIC, NULL, 0, Name, 0, TOCSize);
    for(index = 0; index < TOCData.NumberOfSections; ++index) {
        XRegisterSignature(index + 1, XSIG_FLAG_AUTOMATIC, 0, Name,
            TOCData.Sections[index].Offset,
            TOCData.Sections[index].Size);
    }

    return TRUE;
}

The code used by the tool if none of the signature DLLs or file lists processes a given file is simply:

XRegisterSignature(-1, XSIG_FLAG_NAMED | XSIG_FLAG_AUTOMATIC, NULL, 0, FileName, 0, 0);

A title which includes all of the content signatures within the XBE itself might use a signature DLL such as this to generate an empty signature file:

BOOL CALLBACK CalculateSignatures(Name, Data, Size)
{
    return TRUE;
}

File Lists (not yet supported)

A file list directs the tool to generate non-literal, named signatures over specific byte ranges within specific files. The file list contains entries of the form:

filename: byteranges

filename is the name of the file (wildcards are permitted) for which the byte ranges apply.

byteranges is a comma-separated list of byte ranges in the form "startbyte-endbyte", indicating a byte range within the file to be signed (endpoints are inclusive), or "startbyte-", implying EOF as the endpoint. For example, "0-127,0x80-0xFF,0x100-" is a valid byte range. Additionally, a byte range of "@size" means byte ranges of the specified size through the end of the file.

The first entry within the file which matches the name of the file being process will be the entry that's used to calculate the signatures.

An example of a file list is:

	Image.bmp: 0-
	*.bmp: 0-127, 128-
	*.xip: @0x10000
