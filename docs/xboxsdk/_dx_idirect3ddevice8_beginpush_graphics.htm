IDirect3DDevice8::BeginPush

Allows the caller to write directly to Direct3D's primary push-buffer. BeginPush is most useful for situations where the caller uses the CPU to generate vertex data, and does not wish to incur the function-call overhead associated with using Xbox Begin/End Block, or the copy overhead of IDirect3DDevice8::DrawVerticesUP.

HRESULT BeginPush(
  DWORD Count,
  DWORD **ppPush
);

Parameters

Count
[in] The number of DWORDS to be reserved in the push-buffer. The caller can use less than this amount, but Direct3D will always guarantee that there are at least Count DWORDs free when BeginPush returns. For best performance, Count should be less than the KickOffSize, as specified by IDirect3D8::SetPushBufferSize. The caller must never write more data than space has been reserved for.
ppPush
[out] Address of a pointer to a DWORD. On return, *ppPush points directly into Direct3D's primary push-buffer.

Return Values

If the method succeeds, the return value is S_OK.

Remarks

After calling BeginPush, IDirect3DDevice8::EndPush must be called once the data is completely written. On debug builds, Direct3D will generate a "Nested calls the push-buffer allocator" error if EndPush is not called. No Direct3D functions that write to the push-buffer may be used until EndPush is called.

Direct3D performs no validation on the data written to the push-buffer, even on debug builds. However, if the data is bad, the GPU may generate a "GRAPHICS HARDWARE" error, where "Method" is the method, "Data" is the value written to that method, and "Get" points to a location in the push-buffer that can be as far away as 1 KB past the offending data. When debugging an error, be sure to dump the 1 KB of push-buffer data that precedes the point indicated by "Get", and search the dump data for the values given as the "Method" and "Data".

The GPU may also generate a "GRAPHICS PARSE" error, where "Get" points to the exact location in the push-buffer where the bad data is located.

To be used as a replacement for DrawVerticesUP, the caller needs to set the vertex format using IDirect3DDevice8::SetVertexShader before calling BeginPush. All attributes in the vertex format must be padded DWORD multiples, and the vertex attributes must be specified in the canonical FVF ordering (position followed by weight, normal, diffuse, and so on).

The following example shows how code using BeginPush might look:

// This code assumes that pDevice is a valid pointer
// to an IDirect3DDevice8 interface.

DWORD *pPush; 
DWORD dwordCount = 4 * VertexCount;
pDevice->SetVertexShader( D3DFVF_XYZRHW );

// The "+5" is to reserve enough overhead for the encoding
// parameters. It can safely be more, but no less.
pDevice->BeginPush( dwordCount + 5, &pPush );

// Note that pPush is returned as a pointer to write-combined memory. Writes to
// write-combined memory should be consecutive and in increasing order. Reads
// should be avoided. Additionally, any CPU reads from memory or the L2 cache
// can force expensive partial flushes of the 32-byte write-combine cache.

pPush[0] = D3DPUSH_ENCODE( D3DPUSH_SET_BEGIN_END, 1 );
pPush[1] = D3DPT_TRIANGLELIST;

// NOTE: A maximum of 2047 DWORDs can be specified to D3DPUSH_ENCODE.
//   If there is more than 2047 DWORDs of vertex data, simply split
//   the data into multiple D3DPUSH_ENCODE( D3DPUSH_INLINE_ARRAY )
//   sections.

pPush[2] = D3DPUSH_ENCODE( D3DPUSH_NOINCREMENT_FLAG | D3DPUSH_INLINE_ARRAY, dwordCount );
pPush += 3;

for (i = 0; i < VertexCount; i++)
{
  pPush[0] = *((DWORD*) &MyVertexArray[i].x);
  pPush[1] = *((DWORD*) &MyVertexArray[i].y);
  pPush[2] = *((DWORD*) &MyVertexArray[i].z);
  pPush[3] = *((DWORD*) &MyVertexArray[i].w);
  
  pPush += 4;
}

pPush[0] = D3DPUSH_ENCODE( D3DPUSH_SET_BEGIN_END, 1 );
pPush[1] = 0;
pPush += 2;

pDevice->EndPush( pPush );
