Partnernet

The Xbox Live team provides a network that hosts test servers specifically intended for game development testing. This network is called PartnerNet. Only games approved for Xbox Live have access to PartnerNet. To obtain approval, contact your account manager. Once you have approval, you will be given a title ID to use when building your project for online testing.

All developers with an open Internet connection can test the Xbox SDK samples from the most recent SDK against PartnerNet. These samples expose the full range of the services provided by Xbox Live and are an excellent way to begin preliminary research into online functionality for your title.

If you have trouble connecting to Partnernet, please refer to the Xbox Live page on Xbox Central. There you will find links to the current status for PartnerNet, setup for Xbox consoles to connect against PartnerNet, and troubleshooting information in the event that you can't connect.

Debugging Two Xbox Consoles at Once

You can do this in Microsoft Visual Studio using the following procedure:

  1. Launch Visual Studio.
  2. Load the project you wish to debug.
  3. In the project's Properties dialog box, select the Debugger properties.
  4. Set the Remote Machine option to the name of the first Xbox.
  5. Launch the debugger.
  6. Start a second instance of Visual Studio.
  7. Load the project you wish to debug again.
  8. In the project's Properties dialog box, select the Debugger properties.
  9. Set the Remote Machine option to the name of the second Xbox.
  10. Launch the debugger.

The machine that Visual Studio copies XBEs to is controlled by a separate registry entry at:

HKEY_CURRENT_USER\Software\Microsoft\XboxSDK\XboxName

Testing and Debugging with MatchSim

When creating fake sessions in MatchSim, those sessions never show up in matchmaking searches on the client. It appears to the developer that QoS probing is rejecting those sessions. This is because MatchSim does not support QoS listening for fake sessions. When testing with MatchSim, you will either need to create sessions with real hosts or temporarily disable QoS probing in your client code to prevent rejection of the fake sessions.

Testing and Debugging with NetSim

You can test your title under a limited-bandwidth, high-latency environment with the Network Simulator tool for Xbox. Netsim operates as a packet forwarder that can be configured to add latency, limit bandwidth, or drop packets.

Note    Netsim will not work with system link play.

Some common problems with NetSim:

  • Because NetSim operates as a "virtual" NIC, NetSim requires a separate IP address from that assigned to the PC it is running on.
  • Furthermore, don't use multiple netcards in the NetSim PC. NetSim's "virtual" LAN driver was designed to allow NetSim to run on machines with only a single netcard. If you try to use NetSim on a multi-NIC machine, the virtual LAN driver randomly chooses one to use. This can complicate NetSim setup.
  • It is preferable to assign the Xbox dev kit a higher NetSim address, for example, 10.0.100.2 instead of 10.0.1.2. The source port assigned to the outgoing traffic is determined by the x in 10.0.x.y, and most firewalls block traffic in the lower port range. Using a higher port will increase your chances of the traffic not getting filtered on its way out.
  • NetSim cannot be used to test system link play in games.

Measuring Packet Loss Using the Quality of Service (QoS) APIs

While XNetQoSLookup will return a rough measurement of packets lost in the returned XNQOSINFO structure, this is not an altogether useful way to determine packet loss. Packet loss tends to happen in bursts in the real world, and most games will not want to send out enough probes over a long enough period of time to obtain accurate numbers. The implication is that games should not determine up front the approximate packet loss of a given connection prior to starting a game session across that connection. Games must respond robustly to packet loss as per the TCR and, if necessary to maintain a stable game experience, determine packet loss in an ongoing manner.
