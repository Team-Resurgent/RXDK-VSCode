Writing an AddData Function (XbRC)

After writing the constructor, the next step to extend XbRC is to write the addData function for the class derived from XBRCRawData and call it from the CXBParser::characters member function. It is possible to have element data inside of the element in the XML file. For example, the XDX specification defines an index buffer element as follows: 

<IndexBuffer id="IndexBuffer1">0 1 2 2 1 3</IndexBuffer>

The data "0 1 2 2 1 3" is not set up during the constructor because it is not an attribute but actual element data. When the parser encounters element data, it calls the CXBParser::characters function. This function is also nothing more then a switch statement. It checks the current context's resource type and jumps to the case for that type. If the type has no case statement, it is not an error and the function returns. (Again, by default right now. It would be trivial to make this an error.) If the type does have a case statement, it calls the addData function of the container for that resource type. The reason it's a switch statement and doesn't just call addData automatically (which would work because addData is a virtual function) is to allow for some special handling of the data, if it's needed before calling addData. The addData function for the container parses the data out of the character string handed in. In the case of a cube map, there is no element data and, as such, this function just returns S_OK. In fact, it is never called, because there is no case defined in CXBParser::characters. For something such as index buffers, the addData function would be passed the wide-character string "0 1 2 2 1 3", and it would be up to the code at that point to parse out the indices and store them for the index buffer. The SAX parser can also call the CXBParser::characters function more than once for a given block of data. It is free to break data up into as many chunks as it wants to. Thus, addData must be written to handle being called more than once to handle multiple chunks of data.
