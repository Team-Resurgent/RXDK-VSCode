Xbox Winsock Implementation Notes

For developers familiar with socket programming and other socket implementations, a brief description of the key features and implementation-specific functionality of the Xbox Winsock Libraries is presented here. For detailed information please refer to the rest of the Xbox Winsock documentation.

TCP Implementation Notes

The following table lists features implemented in the Xbox TCP stack.

FeatureReference
Slow StartRFC 2001, RFC 2581
Congestion AvoidanceRFC 2001, RFC 2581
Fast RetransmitRFC 2001, RFC 2581
Fast RecoveryRFC 2001, RFC 2581
Exponential Backoff for Retransmission TimeoutRFC 1072
Silly-Window Syndrome AvoidanceRFC 813

The Xbox TCP stack does not implement the following features:

  • Nagling Coalescence -- A technique in which small packets are combined into larger ones to reduce packet overhead.
  • Selective Acknowledgement (SACK) -- For information on SACK, see RFC 1323 and RFC 2018

Although the Xbox TCP stack does not implement these features, it delays acknowledgements up to 200ms in order to minimize the number that are sent.

Socket Types

The following socket types are supported:

  • SOCK_STREAM (TCP)
  • SOCK_DGRAM (UDP or VDP)

Voice-And-Data Protocol (VDP)

The Xbox Winsock Libraries support the Voice-And-Data Protocol (VDP). The constant, IPPROTO_VDP, represents the protocol number for VDP. The VDP protocol is used with socket type SOCK_DGRAM. To use this protocol, create a socket as follows: 

SOCKET s = socket(AF_INET, SOCK_DGRAM, IPPROTO_VDP);

This socket behaves exactly like a UDP socket, except that messages sent or received on this socket must be in the format:

[cbGameData][GameData][VoiceData]

where [cbGameData] is a two-byte integer in Intel byte order that specifies the number of bytes of game data in the [GameData] part of the message. The remainder of the message [VoiceData] contains voice data that the title chooses to transmit along with the game data. It is acceptable to specify zero bytes of data in [GameData] and/or zero bytes of data in [VoiceData]; a properly formatted message will always be at least two bytes long: enough to hold the [cbGameData] part of the message. A message in which the first two bytes specify more data than is in the entire message is invalid and is not transmitted.

On the wire, only the [GameData] portion of the message is encrypted (with padding). The [VoiceData] portion is transmitted in the clear.

At the receiving end of a connection, a title must receive incoming VDP packets on a socket created to send and receive IPPROTO_VDP. That is, messages of type VDP arrive only on sockets of type VDP. The bind function fails with WSAEADDRINUSE, if a title attempts to bind both a UDP socket and a VDP socket to the same inbound port.

The security gateway code also supports VDP packets. A server specification can have a "Protocol" option whose value is "VDP". Such an inbound port specification causes the security gateway to reject any packets sent to that port that are not of type VDP. For packets that are of type VDP, the security gateway decrypts the packet and constructs a normal UDP packet that contains a message in the above format. The security gateway then sends the UDP packet to the target server on the configured destination port.

Responses from the server originating from the server specification's IP:port address are interpreted as having the format [cbGameData][GameData][VoiceData]. For all ports configured with "Protocol VDP", a game server should be prepared to accept UDP messages in this format, and must reply in the same format. In replies, the security gateway encrypts only the [GameData] portion on the wire.

A server specification that does not specify "Protocol VDP" allows TCP and/or UDP traffic to flow, but blocks inbound VDP traffic.

Socket Modes

Both blocking and nonblocking sockets are supported. By default, sockets are created with the overlapped attribute set. Non-overlapped sockets are not supported. However, you may perform synchronous (non-overlapped) operations by specifying NULL for the lpOverlapped parameter in functions that support that parameter.

Socket I/O models

The two supported input/output (I/O) models are select and overlapped I/O event notification. Message notification, event notification, overlapped I/O completion routines, and I/O completion ports are not supported.

Each socket can have only one pending overlapped I/O send operation and only one pending overlapped I/O receive operation. Thus at most two operations can be pending per socket. The pending send operation can be WSASend or WSASendTo. The pending receive operation can be WSARecv or WSARecvFrom.

Socket Options

The following Winsock options are supported for level SOL_SOCKET:

  • SO_ACCEPTCONN
  • SO_BROADCAST
  • SO_DONTLINGER
  • SO_EXCLUSIVEADDRUSE
  • SO_LINGER
  • SO_RCVBUF
  • SO_RCVTIMEO
  • SO_REUSEADDR
  • SO_SNDBUF
  • SO_SNDTIMEO
  • SO_TYPE

The following Winsock options are supported for level IPPROTO_TCP:

  • TCP_NODELAY

    Note   TCP_NODELAY is accepted, but ignored. The Xbox Secure Network Library (SNL) always does no delay.

No other socket options are supported.

Host Names

gethostbyname and gethostbyaddr are not available for use by titles. Host information is acquired through functions provided by the Xbox Live Library and the Xbox Secure Network Library. For more information, see the Playing Games with Matchmaking, the Xbox Secure Network Library, and XNetXnAddrToInAddr.

Multithreaded Access

Multiple threads cannot simultaneously call Xbox Winsock functions to operate on the same socket. Either a single thread must always be used to access a given socket, or the title must use a synchronization method to serialize thread access to that socket.

When most Xbox Winsock functions are called, the Winsock layer locks access to the socket specified. Until the function returns, other Winsock calls on the same socket will fail. When the function returns, it releases the lock and Winsock calls can again be made on the socket. If a Winsock call is made on a locked socket, the call will fail, returning SOCKET_ERROR, and WSAGetLastError will return WSAEINPROGRESS.

This restriction means that, for example, it is not possible to have one thread receiving from a socket while another thread is writing to that socket.

If a title wants to access the same socket from multiple threads, it must use a synchronization mechanism, such as a critical section or mutex, to serialize access to the socket.

This optimization means that titles that use a single thread to access a particular socket will not incur the overhead of a synchronization object automatically allocated by the system. Only titles that desire multithreaded access will need to allocate such an object themselves, incurring the associated overhead.

Broadcasts and the Local Subnet

When the local Xbox stack sends broadcast packets, neither full broadcast packets (those sent to address 255.255.255.255) nor packets sent to a subnet broadcast address will be received by the local stack, even if the local stack's interface is a part of the subnet being broadcast to.

Multicast is not Supported

Xbox Winsock does not support sending or receiving multicasts.

Outbound Datagrams

Xbox Winsock will not fragment outbound datagrams. The largest payload that may be sent (after overhead is accounted for) is 1304 bytes. Detailed information concerning Xbox packet formats and overhead can be found in Xbox Secure Sockets.

Socket Descriptors and File Handles

Socket descriptors returned by Winsock are usable only in Winsock calls. They are not treated as file handles by the system software. They cannot be used in file handle–based calls such as ReadFile, WriteFile, CloseHandle, and DuplicateHandle, nor in C run-time file routines such as _open, _osfhandle, _read, and _write.
